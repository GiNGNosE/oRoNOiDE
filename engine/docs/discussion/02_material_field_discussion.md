# 決定 2: Per-Voxel マテリアルフィールド設計

**目的**: 各アクティブボクセルに何を格納するかを決定する。この設計はレンダリング・破壊・風化・衝突の全てのサブシステムの能力上限を規定する、基盤中の基盤である。

---

## 1. 先行決定による制約

| 決定 | 内容 | 本決定への影響 |
|------|------|--------------|
| 1a: SBP | カスタム Sparse Brick Pool（SSBO 直接アクセス） | フィールドは std430 レイアウトで GPU バッファに直接格納。テクスチャ経由のアクセスは想定しない |
| 1b: 適応的細分化 | マルチ解像度ブリック | 同一フィールドが複数の解像度で存在する。粗い解像度ではフィールド値の集約方法が必要 |
| 1c: 量子化 | float16 → uint8/int8 に量子化（常時有効） | **量子化後のバイト数**が実際の VRAM コスト。設計は量子化前の論理型で行い、格納は量子化後の物理型 |
| 1c: 法線格納 | SDF 量子化のため法線を別チャネルで格納（snorm8×2 八面体） | コア層に +2B。SDF 勾配計算の精度問題を回避 |

---

## 2. 評価軸

以下の軸で各設計を評価する。**重み**は本プロジェクトの目標（ハイパーリアルな石 + クラフトゲーム）に基づく相対的重要度。

| # | 評価軸 | 説明 | 重み |
|---|--------|------|------|
| E1 | **レンダリング忠実度** | 石の外観がどこまで物理的に正しいか。鉱物混合、結晶方位、風化の連続的表現 | 最高 |
| E2 | **破壊の構造的正確性** | 割れ方が鉱物学的に正しいか。劈開面、異方性、損傷蓄積の精度 | 最高 |
| E3 | **メモリ効率** | 28M ボクセルの風景シーンで VRAM がどれだけ必要か | 高 |
| E4 | **破壊時のランタイムコスト** | 拡張層の展開/解放、フィールド初期化のオーバーヘッド | 高 |
| E5 | **実装の段階的導入** | Phase 1 で最小限で動作し、後のフェーズで拡張できるか | 中 |
| E6 | **永続化の完全性** | セーブ/ロードで全ての状態が復元されるか（損傷、風化進行度等） | 中 |
| E7 | **将来のサブシステム拡張性** | 風化連成（moisture/temperature）、植生 coupling、音響応答への拡張余地 | 中 |

## 3. フィールド要件マップ: 誰が何を必要とするか

各サブシステムが必要とするフィールドとアクセスパターン（R = 読取、W = 書込、- = 不要）。

| フィールド | レンダリング | 破壊シミュレーション | 風化シミュレーション | 衝突判定 | プロシージャル生成 |
|-----------|:-----------:|:-----------------:|:-----------------:|:--------:|:----------------:|
| **SDF** | R | R/W | - | R | - |
| **法線 (snorm8×2)** | R | - | - | R | - |
| **mineral_id** | R | R | R | - | seed |
| **volume_fraction[N]** | R | R | R/W | - | - |
| **porosity** | R (SSS) | R | R/W | - | - |
| **grain_size** | R (texture scale) | R | - | - | seed |
| **weathering_degree** | R (色変化, 薄膜) | R (閾値) | R/W | - | - |
| **microcrack_density** | R (表面粗さ) | R (閾値) | R/W | - | - |
| **damage_d** | R (クラック描画) | R/W | - | - | - |
| **crystal_orientation** | R (異方性反射) | R (劈開面) | - | - | seed |
| **E, nu** | - | R | - | R (反発) | - |
| **Gc, sigma_t** | - | R | R (低下) | - | - |
| **anisotropy_tensor** | - | R | - | - | - |
| **moisture** | - | - | R/W | - | - |
| **temperature** | - | - | R/W | - | - |

### 観察

1. **レンダリング**は mineral_id, volume_fraction, porosity, grain_size, weathering_degree, microcrack_density, crystal_orientation, damage_d, 法線を読む — フィールド数が多い
2. **破壊シミュレーション**は力学パラメータ（E, nu, Gc, sigma_t, anisotropy_tensor）と damage_d を読み書き — 最も重いアクセスパターン
3. **風化**は低頻度だが広範囲のフィールドを徐々に変更する
4. **衝突判定**は SDF + 法線 + E/nu のみ — 軽量
5. **力学パラメータ（E, nu, Gc, sigma_t, anisotropy_tensor）をレンダリングは必要としない** — コア/拡張の分離の根拠

## 4. 候補レイアウト

### 候補 A: コンパクト導出型（Hyperreal 由来）

全ての力学パラメータを mineral_id + porosity から**ルックアップテーブルで導出**する。

```
[全ボクセル]
SDF:                 int8 + header     (1B + ブリックヘッダ)
法線:                snorm8×2          (2B)
mineral_id:          uint8             (1B)
crystal_orientation: snorm8×2          (2B)  ← 八面体エンコード
porosity:            uint8             (1B)
microcrack_density:  uint8             (1B)
weathering_degree:   uint8             (1B)
grain_size:          uint8             (1B)
──────────────────────────────────────
合計: 10 bytes/voxel（量子化後）
```

| 評価軸 | 評価 | 理由 |
|--------|------|------|
| E1 レンダリング忠実度 | **中** | mineral_id が離散 → 粒界でバンディング。volume_fraction なし |
| E2 破壊正確性 | **低** | 力学パラメータが導出のみ → 損傷/風化で乖離した状態を表現不可 |
| E3 メモリ効率 | **最高** | 28M × 10B = 280 MB |
| E4 ランタイムコスト | **最高** | 層管理なし |
| E5 段階的導入 | **高** | 単純 |
| E6 永続化 | **低** | damage_d がない → 損傷がセーブされない |
| E7 拡張性 | **低** | 力学場のオーバーライドが構造的に不可能 |

### 候補 B: 明示的全場（Structural 由来）

全フィールドを全ボクセルに格納する。

```
[全ボクセル]
SDF:                 int8 + header     (1B)
法線:                snorm8×2          (2B)
mineral_id:          uint8             (1B)
volume_fraction[4]:  uint8×4           (4B)
grain_size:          uint8             (1B)
porosity:            uint8             (1B)
crystal_orientation: snorm8×2          (2B)
weathering_degree:   uint8             (1B)
microcrack_density:  uint8             (1B)
damage_d:            uint8             (1B)
E:                   float16           (2B)
nu:                  uint8             (1B)
Gc:                  float16           (2B)
sigma_t:             float16           (2B)
anisotropy_tensor:   snorm8×3          (3B)
moisture:            uint8             (1B)
temperature:         uint8             (1B)
──────────────────────────────────────
合計: 27 bytes/voxel（量子化後）
```

| 評価軸 | 評価 | 理由 |
|--------|------|------|
| E1 レンダリング忠実度 | **最高** | volume_fraction で連続的な鉱物混合、全フィールド即時参照可能 |
| E2 破壊正確性 | **最高** | 明示的力学場 → 損傷/風化後の非標準状態を完全に表現 |
| E3 メモリ効率 | **低** | 28M × 27B = 756 MB |
| E4 ランタイムコスト | **高** | 層管理不要だが、全フィールドの初期化・一貫性維持が重い |
| E5 段階的導入 | **低** | 全フィールドを最初から用意する必要がある |
| E6 永続化 | **最高** | 全状態が per-voxel で永続化 |
| E7 拡張性 | **最高** | 新フィールド追加が容易（レイアウト変更のみ） |

### 候補 C: 二層フィールド（統合案）

レンダリング + 損傷蓄積に必要なフィールドを**コア層**に、破壊シミュレーション固有のフィールドを**拡張層**に分離する。

```
[コア層 — 全アクティブボクセル]
SDF:                 int8 + header     (1B + ブリックヘッダ)
法線:                snorm8×2          (2B)
mineral_id:          uint8             (1B)
volume_fraction[4]:  uint8×4           (4B)
porosity:            uint8             (1B)
grain_size:          uint8             (1B)
weathering_degree:   uint8             (1B)
microcrack_density:  uint8             (1B)
damage_d:            uint8             (1B)
──────────────────────────────────────
コア: 13 bytes/voxel（量子化後）

[拡張層 — 破壊/風化シミュレーション対象領域のみ]
E:                   float16           (2B)  ← 力学計算は量子化しない
nu:                  float16           (2B)
Gc:                  float16           (2B)
sigma_t:             float16           (2B)
anisotropy_tensor:   float16×3         (6B)
moisture:            float16           (2B)
temperature:         float16           (2B)
──────────────────────────────────────
拡張: 18 bytes/voxel（対象領域のみ、float16 維持）
```

| 評価軸 | 評価 | 理由 |
|--------|------|------|
| E1 レンダリング忠実度 | **最高** | volume_fraction + 全レンダリング関連フィールドがコア層に存在 |
| E2 破壊正確性 | **高〜最高** | 拡張層が展開されれば候補 B と同等。展開前は導出値で近似 |
| E3 メモリ効率 | **高** | 28M × 13B = 364 MB（コアのみ）。拡張は操作圏の数個分のみ |
| E4 ランタイムコスト | **中** | 拡張層の展開/解放のライフサイクル管理が必要 |
| E5 段階的導入 | **最高** | Phase 1 はコア層のみで動作。拡張層は破壊実装時に追加 |
| E6 永続化 | **高** | damage_d がコア層にあり常時永続化。拡張層は必要時に再導出可能 |
| E7 拡張性 | **高** | 拡張層にフィールド追加が容易。コア層は安定 |

## 6. 重要論点

以下の各論点は、候補選定の後も個別に設計判断が必要な項目である。

---

### 論点 1: crystal_orientation の扱い — 3 つの方位は統合できるか

**背景**: 石には 3 つの異なる「方向」の物理量がある。

| 物理量 | 何を表すか | 使う場面 | スケール |
|--------|-----------|---------|---------|
| `crystal_orientation` | 結晶粒の主軸方向 | 劈開面の方向決定、異方性反射（グリント） | 粒レベル（~0.1-5mm） |
| `layering_orientation` | 堆積層の法線方向 | 縞模様のテクスチャ方向、層状割れ | 石全体〜領域レベル |
| `anisotropy_tensor` | 力学的な異方性方向 | 破壊伝播の優先方向 | ボクセルレベル |

**議論すべき点**:

- **案 1: crystal_orientation をコア層に格納** (snorm8×2 = 2B)。layering_orientation は per-Entity または per-Region メタデータとして外部化。anisotropy_tensor は拡張層に残す。
  - 根拠: crystal_orientation はレンダリング（異方性反射）と破壊（劈開面）の両方に影響し、ボクセルごとに異なる。layering_orientation は石の中で比較的一様であり per-voxel は過剰
  - コスト: コア層に +2B

- **案 2: anisotropy_tensor をコア層に統合し、他は導出**
  - 根拠: anisotropy_tensor があれば crystal_orientation と layering_orientation の効果を 1 つのフィールドで近似できる
  - 問題: レンダリング用の結晶方位と力学用の異方性は物理的に別の量。統合すると一方の精度が犠牲になる

- **案 3: crystal_orientation のみコア層、anisotropy_tensor は拡張層、layering_orientation は per-Entity**
  - 根拠: 各量を適切な粒度で管理。最も物理的に正確だが、コア +2B

**評価軸への影響**:

| 案 | E1 レンダリング | E2 破壊 | E3 メモリ | 備考 |
|----|:-------------:|:------:|:---------:|------|
| 案 1 | **最高** | 高 | +2B/voxel | layering が per-Entity で粗くなる |
| 案 2 | 中 | 高 | ±0B | 結晶方位の異方性反射が不正確になりうる |
| 案 3 | **最高** | **最高** | +2B/voxel | 最も物理的に正確 |

### 論点 2: volume_fraction の設計 — 固定長 vs スパース

**背景**: 鉱物の混合比（例: 60% 石英 + 30% 長石 + 10% 雲母）を表現するためのフィールド。粒界付近でグラデーションを持ち、バンディングを防ぐ。

**方式**:

| 方式 | 格納 | メモリ | 長所 | 短所 |
|------|------|--------|------|------|
| **固定長配列 uint8×4** | 最大 4 鉱物の混合比を常時格納 | 4B/voxel | 実装が単純。GPU アクセスがストレート | 大半のボクセルは単一鉱物 → [255,0,0,0] で 3B が無駄 |
| **パレット + インデックス** | mineral_id のみ格納。粒界ボクセルのみ volume_fraction を持つ | 1B + スパース追加 | 非粒界では 3B 節約 | ブリック内にスパースマップが必要。GPU アクセスが分岐 |
| **2 鉱物限定 uint8×2** | 主鉱物と副鉱物の 2 つのみ | 2B/voxel | 固定長で GPU フレンドリー。4 鉱物より省メモリ | 3 鉱物以上の遷移帯を表現できない |

**議論すべき点**:

- **固定 4 鉱物は本当に必要か?** 実際の石で 1 つのボクセル（0.1-1mm）に 4 鉱物が混在する状況は粒界ごく近傍のみ。2 鉱物で十分な場合が多い
- **スパース方式は GPU に向かない**: SBP の SSBO 直接アクセスの利点は固定ストライドのアドレッシングにある。ボクセルごとにサイズが異なるとストライドが不定になり、GPU 性能が悪化する
- **2 鉱物限定は妥協として許容可能か?**: mineral_id (主鉱物) + secondary_mineral_id (副鉱物) + blend_factor (uint8) = 3B で、固定 4 鉱物の 4B より 1B 少ない。3 鉱物以上の遷移は稀であり、レンダリング上の差は微小

**評価軸への影響**:

| 方式 | E1 レンダリング | E3 メモリ | E4 ランタイム | 備考 |
|------|:-------------:|:---------:|:------------:|------|
| uint8×4 | **最高** | -4B | 最高（固定ストライド） | 3B の無駄が大半のボクセルで発生 |
| スパース | **最高** | 節約 | 低（分岐） | SBP の設計原則に反する |
| uint8×2 + blend | 高 | -3B | 最高（固定ストライド） | 3 鉱物以上の遷移を切り捨て |

### 論点 3: damage_d の配置 — コア層 vs 拡張層

**背景**: `damage_d` はフェーズフィールド損傷変数（0=無傷, 1=完全破壊）。破壊シミュレーションの核心。

**コア層に置く場合**:
- レンダリングが直接参照可能（クラック描画、表面粗さへの寄与）
- セーブ/ロードで自動的に永続化される
- 破壊が起きていない石でも 1B/voxel のコストが発生
- 大半の石で damage_d = 0（全バイトが 0x00）

**拡張層に置く場合**:
- 破壊が起きていない石のメモリを完全に節約
- レンダリング時に拡張層の存在確認が必要（分岐コスト）
- セーブ時に拡張層の永続化ロジックが必要

**評価軸への影響**:

| 配置 | E1 レンダリング | E3 メモリ | E6 永続化 | 備考 |
|------|:-------------:|:---------:|:---------:|------|
| コア層 | **高**（直接参照） | -1B/voxel | **最高** | 0 が大半だが量子化後 1B の追加コスト |
| 拡張層 | 中（分岐必要） | 節約 | 中（要追加ロジック） | 未破壊の石で完全ゼロコスト |

**補足**: 28M ボクセルで 1B = 28 MB。VRAM 全体（~350MB）の 8%。無視できるほど小さくはないが、レンダリングでの分岐回避と永続化の簡素化を考えると、コア層配置の方が設計的に健全。

### 論点 4: 力学パラメータの初期導出 vs 明示格納

**背景**: E（ヤング率）, nu（ポアソン比）, Gc（破壊靭性）, sigma_t（引張強度）は鉱物の力学的性質。

**導出アプローチ**（候補 A 的）:
```
E = lookup_table[mineral_id].E × f(porosity)
```
- mineral_id + porosity から一意に決まる
- 追加ストレージゼロ
- **問題**: 風化や累積損傷で「本来の鉱物よりも脆くなった」状態を表現できない

**明示格納アプローチ**（候補 B 的）:
```
E = voxel.E  // 直接読み取り
```
- 初期値は mineral_id から導出して設定
- 風化/損傷で値を直接変更可能
- 追加 8B/voxel（E + nu + Gc + sigma_t、全て float16）

**ハイブリッドアプローチ**（候補 C 的）:
```
if (拡張層が存在する):
    E = extension.E  // 明示値が優先
else:
    E = lookup_table[mineral_id].E × f(porosity)  // 導出
```
- 通常時はゼロコスト
- 破壊/風化の対象になった時点で拡張層を展開し、導出値を初期値として書き込む
- 以降は明示値を更新

**議論すべき点**:

- **風化で力学パラメータはどの程度乖離するか?** 風化が進むと porosity が増加し E が低下するが、これは porosity の変化として表現可能。ただし、microcrack の蓄積による Gc の低下は porosity だけでは表現できない — 明示格納が必要になるケース
- **拡張層の展開タイミング**: 衝突が検出された時? 破壊シミュレーション開始時? プレイヤーが「持つ」操作をした時?
- **拡張層の解放タイミング**: シミュレーション完了後、結果をコア層に反映して解放? それとも永続化?

**評価軸への影響**:

| アプローチ | E2 破壊 | E3 メモリ | E4 ランタイム | E5 段階的 |
|-----------|:------:|:---------:|:------------:|:---------:|
| 導出のみ | 低 | **最高** | **最高** | **最高** |
| 明示格納 | **最高** | 低 | 高 | 低 |
| ハイブリッド | 高〜最高 | **高** | 中 | **最高** |

### 論点 5: moisture / temperature の粒度 — per-voxel は必要か

**背景**: 風化シミュレーション（凍結融解サイクル、苔の成長条件判定）に必要。

| 粒度 | メモリ | 精度 | ユースケース |
|------|--------|------|------------|
| **per-voxel** (float16×2 = 4B) | 拡張層に +4B | ボクセル精度の水分分布 | 表面の水が crevice に溜まる、内部への浸透勾配 |
| **per-brick** (float16×2 = 4B) | ブリックヘッダに +4B | ブリック解像度（8^3 ≈ 8mm〜80mm の領域） | 大まかな温度/湿度の空間分布 |
| **per-entity** (float16×2 = 4B) | Entity メタデータに +4B | 石全体で均一 | 「この石は湿っている」程度 |

**議論すべき点**:

- 水分が crevice に溜まる表現は per-voxel が必要だが、これは**風化シミュレーションの頻度が低い**（数秒〜数分ごと）ため、シミュレーション時にのみ一時バッファとして確保し、結果を weathering_degree に反映して解放する方法もある
- temperature は屋外シーンでは空間的に緩やかに変化する → per-entity か per-chunk で十分な場合が多い
- 凍結融解は「表面 ±数 voxel」の温度勾配が重要 → per-voxel が理想的だが、これも一時バッファ方式で対応可能

**推奨**: per-entity を基本とし、風化シミュレーション実行時のみ per-voxel の一時バッファを確保する。永続的に per-voxel で保持するコストに見合わない。

### 論点 6: GPU メモリレイアウト — プレーナー vs インターリーブ

**背景**: ブリック内のボクセルデータを GPU バッファにどう配置するか。Avoyd の分析から、レイアウトがキャッシュ効率に大きく影響することが判明している。

**インターリーブ（AoS: Array of Structures）**:
```
voxel[0]: {SDF, normal, mineral_id, vf[4], porosity, ...}
voxel[1]: {SDF, normal, mineral_id, vf[4], porosity, ...}
...
```
- 1 ボクセルの全フィールドがメモリ上で連続
- 1 ボクセルを完全に読む場合にキャッシュヒット率が高い
- レンダリング（多くのフィールドを同時に読む）に有利

**プレーナー（SoA: Structure of Arrays）**:
```
SDF[0], SDF[1], SDF[2], ...
normal[0], normal[1], normal[2], ...
mineral_id[0], mineral_id[1], ...
```
- 同一フィールドの全ボクセルがメモリ上で連続
- 特定のフィールドだけスキャンする場合にキャッシュヒット率が高い
- 破壊シミュレーション（SDF + damage_d のみ高頻度アクセス）に有利
- フィールドの追加/削除が容易（他チャネルに影響しない）

**議論すべき点**:

- レンダリングパスでは SDF + 法線 + mineral_id + volume_fraction + weathering_degree + ... の**多チャネル同時読み取り**が発生 → AoS 有利
- 破壊パスでは SDF + damage_d の**2 チャネル集中アクセス**が発生 → SoA 有利
- Phase 1-2 では NanoVDB 変換を経由するため、SBP のレイアウトは CPU 側のみに影響。Phase 3-4 で GPU 直接アクセスになった時にレイアウトが性能を直接支配する
- **両方のパスを最適化するのは困難** — どちらかを優先する必要がある

**推奨**: Phase 1 では SoA（フィールド追加が容易、柔軟性優先）。Phase 3-4 でプロファイリング後に AoS への移行を検討。

### 論点 7: 解像度間のフィールド集約方法

**背景**: 適応的細分化（決定 1b）により、同じ石が近距離では 0.25mm/voxel、遠距離では 5mm/voxel で表現される。粗い解像度のボクセルは、細かい解像度の複数ボクセルを「代表」する必要がある。

| フィールド | 集約方法 | 理由 |
|-----------|---------|------|
| SDF | 再サンプリング（trilinear） | 距離場は空間的に滑らか |
| 法線 | 再計算（粗い SDF から） | 集約した SDF から再計算する方が正確 |
| mineral_id | **多数決** or **支配的鉱物** | 離散値は平均化できない |
| volume_fraction | **加重平均** | 連続値。混合比の空間平均 |
| porosity | 加重平均 | 連続値 |
| grain_size | 加重平均 | 連続値 |
| weathering_degree | 加重平均 | 連続値 |
| microcrack_density | **最大値** | 保守的に最も損傷の大きい値を採用（粗い LOD でクラックが消えないように） |
| damage_d | **最大値** | 同上 |

**議論すべき点**:

- mineral_id の多数決で情報が失われる → volume_fraction があれば粗い LOD でも混合比で表現可能。これは volume_fraction をコア層に含める追加の根拠
- 粗化時にフィールド集約を行うタイミング: 粗化イベント発生時に一括計算? それともブリック生成時に遅延計算?
- 粗化→細分化→粗化のラウンドトリップでフィールド値が劣化しないか（不可逆な集約の蓄積）
