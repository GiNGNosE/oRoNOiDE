# 決定 2: Per-Voxel マテリアルフィールド設計

**目的**: 各アクティブボクセルに何を格納するかを決定する。この設計はレンダリング・破壊・風化・衝突の全てのサブシステムの能力上限を規定する、基盤中の基盤である。

---

## 1. 先行決定による制約

| 決定 | 内容 | 本決定への影響 |
|------|------|--------------|
| 1a: SBP | カスタム Sparse Brick Pool（SSBO 直接アクセス） | フィールドは std430 レイアウトで GPU バッファに直接格納。テクスチャ経由のアクセスは想定しない |
| 1b: 適応的細分化 | マルチ解像度ブリック | 同一フィールドが複数の解像度で存在する。粗い解像度ではフィールド値の集約方法が必要 |
| 1c: 量子化 | float16 → uint8/int8 に量子化（常時有効） | **量子化後のバイト数**が実際の VRAM コスト。設計は量子化前の論理型で行い、格納は量子化後の物理型 |
| 1c: 法線格納 | SDF 量子化のため法線を別チャネルで格納（snorm8×2 八面体） | コア層に +2B。SDF 勾配計算の精度問題を回避 |

## 2. 設計哲学: 戦略的妥協によるリアリティ

完全な物理シミュレーションは不可能である — これは前提。目標は**知覚的リアリズム**であり、物理的正確性の網羅ではない。設計の中心的問いは:

> **どこで、どのように妥協すれば、観察者にとって現実と区別がつかなくなるか?**

この問いに対する回答が**不可逆/可逆の分離**である。これはメモリ最適化ではなく、リアリティと実用性を両立させるための**妥協戦略**:

- **不可逆な現象が石の「見た目」を決定する。** 入ったクラック、進行した風化、鉱物構造そのもの — これらは観察者が見る永久的な記録。これらが忠実に格納されている限り（コア層）、シミュレーションが動いていなくても石は正しく見える。
- **可逆な現象はアクティブイベント中にしか観測できない。** 衝撃中の弾性変形、石の中を伝播する応力場、蒸発する水分 — これらは一時的。地面に置かれた石に応力場があるかどうかを観察者は判別できない。従って、必要時に計算し、完了後に解放しても知覚的リアリティは一切損なわれない。

**核心的洞察**: 10 秒前にクラックが入って今は静止している石は、アクティブにシミュレーション中の石と**見た目が完全に同一**である — なぜなら、全ての永続的結果（microcrack_density、weathering_degree、SDF トポロジー変更）は既にコア層にコミットされている。拡張層が存在しないことは視覚品質を一切劣化させない。

これは MECE のフィルター基準も提供する: **per-voxel のコアフィールドが正当化されるのは、永続的な視覚的または構造的痕跡を残す現象をカバーする場合のみ。** アクティブイベント中にしか観測できない現象は、拡張層でオンデマンドにカバーすれば十分。

### 3.5 MECE ギャップチェック

**導出でカバー済み（別途フィールド不要）**: V2（粗さ）, V3（金属）, V4（SSS）, S1（弾性）, S2（破壊開始）, S4（靭性）, I1-I4 — 全てコアフィールドから LUT 経由で導出可能

**温度関連（新規追加）**: V11（熱放射）, W5（熱衝撃）, I5（熱伝達）— per-Entity temperature + シェーダ/シミュレーションで対応。W5 のみ per-voxel temperature（拡張層、一時的）が必要。詳細は Section 3.7 で設計

**要検討**: V6（layering_orientation）— per-Entity で十分か per-voxel が必要か

**SRP 検証が必要**: V9（クラック描画）と S7（フェーズフィールド）と W3（損傷蓄積）— damage_d / microcrack_density の重複問題

### 3.6 ユースケース駆動 MECE 検証

現象ベースの MECE（3.1–3.5）は「物理的に何が起こるか」を網羅する。ここではそれを補完し、「**プレイヤーが何をするか**」の視点から、フィールド設計が実際のゲームプレイ要件を満たしているかを検証する。

#### 代表的アクション（頻出）

| アクション | 期待される観察可能な応答 | 必要なフィールド / システム | カバー状況 |
|-----------|------------------------|--------------------------|-----------|
| **投げる** | 放物線飛行 → 着地時に衝突音、地面に跳ね返りまたは停止 | 質量（導出: mineral_id + porosity）, E/nu（導出: 反発係数）, SDF（衝突形状） | **カバー済み** — I1, I3, I4 |
| **叩く / 割る** | 衝撃部位にクラック発生、閾値超えで破片分離、破断面の内部構造が露出 | phase_field_d（拡張）, sigma_t/Gc（導出）, crystal_orientation, mineral_id, microcrack_density, SDF トポロジー変更 | **カバー済み** — S2-S6, V9 |
| **積む / 建築** | 安定なスタッキング、接触面の摩擦で滑らない | SDF（接触検出）, 摩擦係数（導出: grain_size + weathering） | **カバー済み** — I2 |
| **拾う / 置く** | オブジェクト選択、持ち上げ・配置、ハイライト表示 | Entity ID, SDF（ピッキング）, instance_id（ECS） | **カバー済み** — ECS + ID 体系（決定 4） |

| **水中に落とす** | 着水エフェクト → 水中で色が暗くなる、引き上げ後に徐々に乾く | moisture（拡張層で per-voxel 一時展開 → 表面浸透勾配）, per-Entity moisture state | **カバー済み** — 論点 5 の moisture 設計で対応 |

#### 境界アクション（低頻度だがリアリティ判定の境界線）

| アクション | 期待される観察可能な応答 | 必要なフィールド / システム | カバー状況 |
|-----------|------------------------|--------------------------|-----------|
| **火に入れる / 窯で加熱** | 温度上昇 → 黒体輻射で赤熱〜白熱に発光。水分蒸発（蒸気エフェクト）。長時間で鉱物変化の可能性 | per-Entity temperature（上昇）, moisture → 0（蒸発）, シェーダで emissive 項（黒体スペクトル） | **要設計** → Section 3.7 |
| **熱い石を水で急冷** | 蒸気爆発エフェクト。温度勾配 → 熱応力 → クラック生成（熱衝撃） | per-voxel temperature（拡張層に一時展開）→ 温度勾配 → 応力場 → microcrack_density 増加 | **要設計** → Section 3.7 |
| **長期間の風雨曝露** | 表面の徐々な変色、微細クラック増加、苔が生える条件の整備 | weathering_degree↑, microcrack_density↑（風化エージェント）, moisture（苔成長条件） | **カバー済み** — W1-W4 |
| **水中に長時間放置** | 色が暗化（水吸収）、表面の化学風化が加速 | moisture per-voxel（拡張層で浸透シミュレーション）, weathering_degree↑ | **カバー済み** — W1 + moisture 設計 |
| **熱い石を他のオブジェクトに接触** | 接触面の焦げ、隣接オブジェクトの温度上昇 | per-Entity temperature, 接触検出, **相手オブジェクト側のマテリアル応答** | **部分カバー** — I5 で設計。石側のフィールド追加は不要、接触システム側の実装 |

#### ギャップ分析

**完全にカバー済み**: 代表的アクション 7 種は全て既存のフィールド設計で対応可能。

**設計が必要な境界アクション**:

1. **加熱 / 冷却 / 熱衝撃**: temperature の扱いが論点 5 で「per-entity 基本」と推奨されているが、熱衝撃シナリオでは per-voxel の温度勾配が不可欠。→ **Section 3.7 で温度モデルを設計**
2. **隣接への熱伝達**: 石のフィールド設計には影響しないが、ゲームシステムとして接触時の熱伝達ロジックが必要

**フィールド設計への影響**:
- **新規フィールド追加は不要**。temperature は既に per-Entity で計画済み、per-voxel 展開は拡張層の既存メカニズムで対応
- **不可逆/可逆分離の原則に完全に適合**: 加熱→冷却のサイクル中に per-voxel temperature を拡張層で使い、結果（microcrack_density の増加）をコアにコミットして拡張層を解放する

### 3.7 温度モデル設計

#### 3.7.1 設計方針

温度は他の物理量と異なり、**空間的に均一な場合が大半**（石全体が同じ気温）で、**局所勾配が意味を持つのは特殊イベント時のみ**（片面だけ火に接している、急冷時）。この特性を活かし、3 層の段階的精度モデルを設計する。

#### 3.7.2 3 層温度モデル

| 層 | 格納先 | 型 | 常時コスト | 用途 |
|----|--------|-----|-----------|------|
| **Layer 1: per-Entity temperature** | Entity メタデータ | float32 | 4B / Entity | 石全体の基本温度。全ての石が常に持つ。環境温度との差分で冷却/加熱を計算 |
| **Layer 2: 導出熱物性** | 格納なし（LUT + コアフィールドから計算） | — | 0B | 比熱容量 c_p、熱伝導率 κ。mineral_id + porosity から LUT で導出 |
| **Layer 3: per-voxel temperature** | 拡張層 SSBO | uint8 or float16 | 0B (非アクティブ時) | 局所温度勾配が必要な場合のみ確保。熱衝撃、片面加熱シナリオ |

#### 3.7.3 冷却/加熱モデル

ニュートンの冷却法則に基づく per-Entity 温度更新:

```
dT/dt = -(h · A / (m · c_p)) · (T - T_env)

h   : 熱伝達係数（環境依存: 空気中 ≈ 10, 水中 ≈ 500–1000 W/m²K）
A   : 表面積（SDF から導出可能。精密計算不要、バウンディング近似で十分）
m   : 質量（= 密度 × 体積。密度は mineral_id + porosity から導出）
c_p : 比熱容量（mineral_id LUT。花崗岩 ≈ 790 J/kgK, 玄武岩 ≈ 840 J/kgK）
T_env: 環境温度（ワールド状態として定義）
```

**比熱 c_p の役割**: 比熱が高い石ほど温まりにくく冷めにくい。これにより同じ火に入れても花崗岩と大理石で加熱速度が異なり、冷却時間も変わる。比熱を考慮しないと「全ての石が同じペースで冷める」という非リアルな挙動になる。

**毎フレームの計算コスト**: per-Entity なので石の個数分（数百〜数千）のスカラー演算のみ。ゲームループへの影響は無視できる。

#### 3.7.4 per-voxel 温度の発動条件と生存期間

per-voxel temperature はコストが高い（拡張層 +1B〜2B/voxel）ため、必要な場合にのみ確保する:

**発動条件**:
- 石の一部のみが熱源に接触（温度勾配が空間的に不均一）
- 急速冷却イベント（per-Entity 温度の変化率が閾値を超えた場合）
- 手動トリガー（ゲームイベントシステムから）

**生存期間**:
```
局所加熱検出
  → 拡張層に per-voxel temperature バッファ確保
  → per-Entity temperature を全 voxel に初期値としてコピー
  → 熱伝導方程式（ラプラシアン）を per-voxel で数ステップ解く
  → 温度勾配 → 熱応力場 → phase_field_d ソルバー
  → 結果: microcrack_density 増加をコアにコミット
  → 石全体の温度が均一化（勾配が閾値以下になった）
  → per-voxel temperature バッファを解放
  → per-Entity temperature のみに戻る
```

**解放判定**: `max(T_voxel) - min(T_voxel) < ε` （石内の温度差が十分小さい）→ per-Entity 温度に集約して解放

#### 3.7.5 レンダリング統合

| 温度範囲 | 視覚効果 | 実装 |
|----------|---------|------|
| < 500°C | 視覚的変化なし | — |
| 500–700°C | 暗赤色の発光 | emissive 項に黒体スペクトル加算（Planck 分布近似） |
| 700–1200°C | 橙〜黄色の発光 | 同上、強度と色温度が上昇 |
| > 1200°C | 白熱 | 同上 |
| 急冷直後 | 蒸気エフェクト + クラック音 | パーティクル + SE。temperature 変化率から発火 |

**シェーダ側の実装**: per-Entity temperature をユニフォームまたは per-instance バッファで渡す。per-voxel 温度がアクティブな場合は拡張層バッファからサンプル。黒体輻射色は `T → (R, G, B)` の近似関数（Tanner Helland approximation 等）で計算し、emissive に加算。格納フィールドの追加は不要。

#### 3.7.6 不可逆/可逆分離との整合

温度モデルは設計原則に完全に適合する:

| 要素 | 分類 | 根拠 |
|------|------|------|
| per-Entity temperature | **per-Entity（可逆）** | 環境温度に向かって必ず収束する。永続的痕跡を残さない |
| 比熱 c_p, 熱伝導率 κ | **導出** | mineral_id + porosity LUT。格納不要 |
| per-voxel temperature | **拡張層（可逆/一時的）** | 勾配が消滅したら解放。情報損失なし |
| 熱衝撃 → microcrack_density | **コア（不可逆）** | 熱衝撃の永続的結果。コアにコミットされる |

**核心**: 火に入れて冷ました石は、microcrack_density の増加という永続的痕跡がコアに残る。温度自体は環境温度に戻り、per-voxel バッファは解放される。観察者が見る「焼かれた石」のリアリティは microcrack_density（+ 将来的には焼け跡テクスチャ用の weathering_degree 変化）がコアに存在することで保証される。

## 4. SRP: フィールド責務一意性監査

MECE で「何をカバーすべきか」を確認した上で、各フィールドが**1 つの物理的問いにのみ答えているか**を検証する。

### 4.1 責務監査テーブル

各フィールドに対して「このフィールドが答える唯一の物理的問い」を定義する。

| フィールド | 唯一の問い | 可変? | 他から導出可能? | 判定 |
|-----------|-----------|:-----:|:--------------:|------|
| SDF | 「表面からの距離は?」 | Yes（トポロジー変更） | No | **コア**（形状） |
| 法線 | 「表面の向きは?」 | 導出可能 | Yes（SDF 勾配） | **コア**（性能最適化、決定 1c で確定済み） |
| mineral_id | 「何の鉱物か?」 | No（内在的） | No | **コア**（同一性） |
| volume_fraction | 「鉱物の混合比は?」 | No（内在的） | No | **コア**（mineral_id とは別: 比率 vs 同一性） |
| porosity | 「空隙率は?」 | 緩やかに可変 | No | **コア**（weathering_degree とは独立） |
| grain_size | 「結晶粒の大きさは?」 | No（内在的） | No | **コア**（内在的性質） |
| crystal_orientation | 「結晶軸の方向は?」 | No（内在的） | No | **コア**（レンダリング + 物理の両方で使用） |
| weathering_degree | 「化学的にどれだけ変質したか?」 | 不可逆 | No | **コア**（porosity とは独立: 化学 vs 空隙形状） |
| microcrack_density | 「機械的にどれだけ損傷したか?」 | 不可逆 | No | **コア**（唯一の永続的損傷指標） |
| ~~damage_d~~ | ~~「ソルバーの損傷状態は?」~~ | 一時的 | — | **冗長 → 拡張層に phase_field_d として移動** |
| E, nu, Gc, sigma_t | 「現在の力学特性は?」 | 再導出可能 | Yes（mineral_id + porosity + microcrack_density） | **拡張層**（シミュレーション時のみ展開） |
| anisotropy_tensor | 「破壊異方性の方向は?」 | 再導出可能 | Yes（crystal_orientation + mineral_id + volume_fraction） | **拡張層**（シミュレーション時に導出） |
| moisture | 「どれだけ湿っているか?」 | 可逆 | No | **拡張層 or per-Entity** |
| temperature | 「温度は?」 | 可逆 | No | **per-Entity** |

### 4.2 重複ペア検証

**damage_d vs microcrack_density**: 同一の物理現象（クラックによる材料劣化）を異なる抽象レベルで表現している。damage_d はフェーズフィールドソルバーの数学的抽象。microcrack_density は観測可能な物理量。ソルバー完了後、damage_d の永続的効果は以下に吸収される:
- SDF トポロジー変更（ボクセルが完全に割れた場合、d=1）
- microcrack_density の増加（応力を受けたが完全には割れなかった場合）

→ **結論**: damage_d は一時的なソルバー変数であり、永続的フィールドではない。microcrack_density が唯一の損傷指標としてコアに残る。

**crystal_orientation vs anisotropy_tensor**: crystal_orientation は内在的な観測量（結晶軸）。anisotropy_tensor は有効力学的結果で、crystal_orientation + 鉱物特性から導出可能。→ **結論**: crystal_orientation をコアに保持。anisotropy_tensor はシミュレーション時に拡張層で計算。

**weathering_degree vs porosity**: 風化は空隙率を増加させるが、表面化学（色、薄膜）、粗さ、鉱物安定性にも独立に影響する。新鮮な多孔質の軽石（高 porosity、低 weathering）と風化した緻密な花崗岩（低 porosity、高 weathering）は物理的に区別可能。→ **結論**: 冗長ではない。両方必要。

### 4.3 SRP 違反チェック

**crystal_orientation は 2 つの問いに答えているのでは?** レンダリング（異方性反射）と物理（劈開面方向）の両方で使用される。しかしこれは同一の物理量（結晶軸方向）を異なるサブシステムが参照しているだけであり、SRP 違反ではない。「結晶軸の方向は?」という 1 つの問いに対して、レンダリングも物理も同じ答えを使う。

**microcrack_density は V9（クラック描画）と W3（損傷蓄積）の両方をカバーしているのでは?** これも同一の物理量。クラックが存在すること（描画に影響）と、クラックが蓄積していること（力学に影響）は同じ現象の同じ側面。SRP 違反ではない。

## 5. 設計原則: 不可逆/可逆分離

MECE と SRP の結論をエンジニアリングルールとして形式化する:

**ルール**: 
- **コア層** = 不可逆的または内在的な状態。「このボクセルが何であるか」「何が永続的に起きたか」を定義する。常に存在し、常に永続化される。**観察者が見るものはここにある** — リアリティはコアに宿る。
- **拡張層** = 可逆的、一時的、または導出可能な状態。コアフィールドから再計算可能であり、シミュレーション完了後に解放しても**情報損失がない**。拡張層が不在でもリアリティは劣化しない — 影響を受けるのはアクティブなシミュレーション能力のみ。

**妥協戦略の実動作**:
```
遠方の石（シミュレーションなし）: コアのみ → 完璧にリアルに見える
衝撃を受けた石: コア + 拡張層 → 物理シミュレーションがフル精度で実行される
衝撃完了後:
  拡張層の結果をコアにコミット（microcrack_density↑, SDF 変更）
  → 拡張層を解放
  → 石は「シミュレーションなしで完璧にリアル」な状態に戻る
```

リアリティの VRAM コストはコア層のみで決まる。シミュレーションコストは pay-as-you-go。

### フィールド分類

| 分類 | フィールド | 根拠 |
|------|-----------|------|
| **コア（不可逆/内在的）** | SDF, 法線, mineral_id, volume_fraction, porosity, grain_size, crystal_orientation, weathering_degree, microcrack_density | 永続的な視覚的/構造的痕跡を残す |
| **拡張（可逆/一時的/導出可能）** | phase_field_d, E, nu, Gc, sigma_t, anisotropy_tensor, moisture, **per-voxel temperature** | シミュレーション中のみ必要、コアから再導出可能。per-voxel temperature は局所温度勾配シミュレーション時のみ確保 |
| **per-Entity** | temperature, layering_orientation, topology_version | 空間的に緩やか or 石全体で均一 |
| **導出（格納なし）** | 比熱容量 c_p, 熱伝導率 κ, 密度, 表面粗さ, 弾性率 E, 破壊靭性 Gc | mineral_id + porosity (+ microcrack_density) から LUT で計算 |

1. **レンダリングが参照するのはコア層のみ** — 拡張層が存在しなくても視覚品質は劣化しない

5. **コア層だけで完全なレンダリングが可能**という性質が、不可逆/可逆分離の設計原則を裏付ける

## 10. 重要論点

以下の各論点は、候補選定の後も個別に設計判断が必要な項目である。

---

### 論点 1: crystal_orientation の扱い — 3 つの方位は統合できるか

**背景**: 石には 3 つの異なる「方向」の物理量がある。

| 物理量 | 何を表すか | 使う場面 | スケール |
|--------|-----------|---------|---------|
| `crystal_orientation` | 結晶粒の主軸方向 | 劈開面の方向決定、異方性反射（グリント） | 粒レベル（~0.1-5mm） |
| `layering_orientation` | 堆積層の法線方向 | 縞模様のテクスチャ方向、層状割れ | 石全体〜領域レベル |
| `anisotropy_tensor` | 力学的な異方性方向 | 破壊伝播の優先方向 | ボクセルレベル |

**議論すべき点**:

- **案 1: crystal_orientation をコア層に格納** (snorm8×2 = 2B)。layering_orientation は per-Entity または per-Region メタデータとして外部化。anisotropy_tensor は拡張層に残す。
  - 根拠: crystal_orientation はレンダリング（異方性反射）と破壊（劈開面）の両方に影響し、ボクセルごとに異なる。layering_orientation は石の中で比較的一様であり per-voxel は過剰
  - コスト: コア層に +2B

- **案 2: anisotropy_tensor をコア層に統合し、他は導出**
  - 根拠: anisotropy_tensor があれば crystal_orientation と layering_orientation の効果を 1 つのフィールドで近似できる
  - 問題: レンダリング用の結晶方位と力学用の異方性は物理的に別の量。統合すると一方の精度が犠牲になる

- **案 3: crystal_orientation のみコア層、anisotropy_tensor は拡張層、layering_orientation は per-Entity**
  - 根拠: 各量を適切な粒度で管理。最も物理的に正確だが、コア +2B

**評価軸への影響**:

| 案 | E1 レンダリング | E2 破壊 | E3 メモリ | 備考 |
|----|:-------------:|:------:|:---------:|------|
| 案 1 | **最高** | 高 | +2B/voxel | layering が per-Entity で粗くなる |
| 案 2 | 中 | 高 | ±0B | 結晶方位の異方性反射が不正確になりうる |
| 案 3 | **最高** | **最高** | +2B/voxel | 最も物理的に正確 |

---

### 論点 2: volume_fraction の設計 — 固定長 vs スパース

**背景**: 鉱物の混合比（例: 60% 石英 + 30% 長石 + 10% 雲母）を表現するためのフィールド。粒界付近でグラデーションを持ち、バンディングを防ぐ。

**方式**:

| 方式 | 格納 | メモリ | 長所 | 短所 |
|------|------|--------|------|------|
| **固定長配列 uint8×4** | 最大 4 鉱物の混合比を常時格納 | 4B/voxel | 実装が単純。GPU アクセスがストレート | 大半のボクセルは単一鉱物 → [255,0,0,0] で 3B が無駄 |
| **パレット + インデックス** | mineral_id のみ格納。粒界ボクセルのみ volume_fraction を持つ | 1B + スパース追加 | 非粒界では 3B 節約 | ブリック内にスパースマップが必要。GPU アクセスが分岐 |
| **2 鉱物限定 uint8×2** | 主鉱物と副鉱物の 2 つのみ | 2B/voxel | 固定長で GPU フレンドリー。4 鉱物より省メモリ | 3 鉱物以上の遷移帯を表現できない |

**議論すべき点**:

- **固定 4 鉱物は本当に必要か?** 実際の石で 1 つのボクセル（0.1-1mm）に 4 鉱物が混在する状況は粒界ごく近傍のみ。2 鉱物で十分な場合が多い
- **スパース方式は GPU に向かない**: SBP の SSBO 直接アクセスの利点は固定ストライドのアドレッシングにある。ボクセルごとにサイズが異なるとストライドが不定になり、GPU 性能が悪化する
- **2 鉱物限定は妥協として許容可能か?**: mineral_id (主鉱物) + secondary_mineral_id (副鉱物) + blend_factor (uint8) = 3B で、固定 4 鉱物の 4B より 1B 少ない。3 鉱物以上の遷移は稀であり、レンダリング上の差は微小

**評価軸への影響**:

| 方式 | E1 レンダリング | E3 メモリ | E4 ランタイム | 備考 |
|------|:-------------:|:---------:|:------------:|------|
| uint8×4 | **最高** | -4B | 最高（固定ストライド） | 3B の無駄が大半のボクセルで発生 |
| スパース | **最高** | 節約 | 低（分岐） | SBP の設計原則に反する |
| uint8×2 + blend | 高 | -3B | 最高（固定ストライド） | 3 鉱物以上の遷移を切り捨て |

### 論点 3: ~~damage_d の配置~~ — **SRP により解決済み**

> **解決**: SRP 分析（セクション 4.2）により、damage_d と microcrack_density は同一の物理現象（材料劣化）を異なる抽象レベルで表現していることが確認された。
>
> - **microcrack_density**（コア層、uint8）: 唯一の永続的損傷指標。レンダリング（クラック描画、表面粗さ）と物理（閾値）の両方で参照。不可逆。
> - **phase_field_d**（拡張層、float16）: ソルバーの一時的作業変数。シミュレーション完了後に結果を microcrack_density と SDF にコミットし、解放。
>
> damage_d という独立した永続フィールドは不要。

### 論点 4: ~~力学パラメータの導出 vs 格納~~ — **設計原則により解決済み**

> **解決**: 不可逆/可逆分離の設計原則（セクション 5）により決定。
>
> - 力学パラメータ（E, nu, Gc, sigma_t）はコアフィールド（mineral_id + porosity + microcrack_density）から**導出可能**な可逆的状態。
> - 通常時: `E = LUT[mineral_id].E × g(porosity) × h(microcrack_density)` — 追加ストレージゼロ
> - シミュレーション時: 拡張層を展開し、導出値を初期値として float16 で格納。ソルバーが更新。
> - シミュレーション完了時: 結果をコアフィールド（microcrack_density↑, porosity↑, SDF 変更）にコミットし、拡張層を解放。
>
> **microcrack_density がコア層にある**ことが鍵: 風化/損傷による Gc の低下は `h(microcrack_density)` で反映され、明示格納なしでも「本来より脆くなった」状態を表現可能。

### 論点 5: moisture / temperature の粒度 — per-voxel は必要か

> **temperature については Section 3.7 で詳細設計済み。** 以下の議論は moisture に焦点を移す。temperature の結論: per-Entity 基本 + per-voxel 拡張層（一時的、熱衝撃/局所加熱時のみ）。

**背景**: 風化シミュレーション（凍結融解サイクル、苔の成長条件判定）に moisture が必要。

| 粒度 | メモリ | 精度 | ユースケース |
|------|--------|------|------------|
| **per-voxel** (float16 = 2B) | 拡張層に +2B | ボクセル精度の水分分布 | 表面の水が crevice に溜まる、内部への浸透勾配 |
| **per-brick** (float16 = 2B) | ブリックヘッダに +2B | ブリック解像度（8^3 ≈ 8mm〜80mm の領域） | 大まかな湿度の空間分布 |
| **per-entity** (float32 = 4B) | Entity メタデータに +4B | 石全体で均一 | 「この石は湿っている」程度 |

**議論すべき点**:

- 水分が crevice に溜まる表現は per-voxel が必要だが、これは**風化シミュレーションの頻度が低い**（数秒〜数分ごと）ため、シミュレーション時にのみ一時バッファとして確保し、結果を weathering_degree に反映して解放する方法もある
- 凍結融解は「表面 ±数 voxel」の温度勾配が重要 → per-voxel temperature で対応済み（Section 3.7）
- 水中に落とした石の色変化は per-voxel moisture が一時的に必要 → 拡張層で展開、乾燥後に解放

**推奨**: temperature と同じ設計パターン — per-Entity moisture を基本とし、浸透/風化シミュレーション時のみ per-voxel 一時バッファを確保。結果（weathering_degree↑, porosity↑）をコアにコミットして解放。

### 論点 6: GPU メモリレイアウト — プレーナー vs インターリーブ

**背景**: ブリック内のボクセルデータを GPU バッファにどう配置するか。Avoyd の分析から、レイアウトがキャッシュ効率に大きく影響することが判明している。

**インターリーブ（AoS: Array of Structures）**:
```
voxel[0]: {SDF, normal, mineral_id, vf[4], porosity, ...}
voxel[1]: {SDF, normal, mineral_id, vf[4], porosity, ...}
...
```
- 1 ボクセルの全フィールドがメモリ上で連続
- 1 ボクセルを完全に読む場合にキャッシュヒット率が高い
- レンダリング（多くのフィールドを同時に読む）に有利

**プレーナー（SoA: Structure of Arrays）**:
```
SDF[0], SDF[1], SDF[2], ...
normal[0], normal[1], normal[2], ...
mineral_id[0], mineral_id[1], ...
```
- 同一フィールドの全ボクセルがメモリ上で連続
- 特定のフィールドだけスキャンする場合にキャッシュヒット率が高い
- 破壊シミュレーション（SDF + phase_field_d のみ高頻度アクセス）に有利
- フィールドの追加/削除が容易（他チャネルに影響しない）

**議論すべき点**:

- レンダリングパスでは SDF + 法線 + mineral_id + volume_fraction + weathering_degree + ... の**多チャネル同時読み取り**が発生 → AoS 有利
- 破壊パスでは SDF + phase_field_d の**2 チャネル集中アクセス**が発生 → SoA 有利（ただし拡張層は別バッファのため、コア層のレイアウトとは独立）
- Phase 1-2 では NanoVDB 変換を経由するため、SBP のレイアウトは CPU 側のみに影響。Phase 3-4 で GPU 直接アクセスになった時にレイアウトが性能を直接支配する
- **両方のパスを最適化するのは困難** — どちらかを優先する必要がある

**推奨**: Phase 1 では SoA（フィールド追加が容易、柔軟性優先）。Phase 3-4 でプロファイリング後に AoS への移行を検討。

### 論点 7: 解像度間のフィールド集約方法

**背景**: 適応的細分化（決定 1b）により、同じ石が近距離では 0.25mm/voxel、遠距離では 5mm/voxel で表現される。粗い解像度のボクセルは、細かい解像度の複数ボクセルを「代表」する必要がある。

| フィールド | 集約方法 | 理由 |
|-----------|---------|------|
| SDF | 再サンプリング（trilinear） | 距離場は空間的に滑らか |
| 法線 | 再計算（粗い SDF から） | 集約した SDF から再計算する方が正確 |
| mineral_id | **多数決** or **支配的鉱物** | 離散値は平均化できない |
| volume_fraction | **加重平均** | 連続値。混合比の空間平均 |
| porosity | 加重平均 | 連続値 |
| grain_size | 加重平均 | 連続値 |
| weathering_degree | 加重平均 | 連続値 |
| microcrack_density | **最大値** | 保守的に最も損傷の大きい値を採用（粗い LOD でクラックが消えないように） |

**議論すべき点**:

- mineral_id の多数決で情報が失われる → volume_fraction があれば粗い LOD でも混合比で表現可能。これは volume_fraction をコア層に含める追加の根拠
- 粗化時にフィールド集約を行うタイミング: 粗化イベント発生時に一括計算? それともブリック生成時に遅延計算?
- 粗化→細分化→粗化のラウンドトリップでフィールド値が劣化しないか（不可逆な集約の蓄積）
