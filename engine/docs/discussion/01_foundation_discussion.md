## 決定0: 設計原則と構造不変条件

### 背景

個別の技術を選定する前に、全ての決定を貫く設計原則を明文化する。Structural レポートはこれを明確に定義しており（Section 9）、技術選定時の「北極星」として機能する。

### 設計原則

1. **単一権威状態**: 形状・材質・物性・破壊履歴は一つのボリュメトリック場に集約する。メッシュ、テクスチャ、コリジョンシェイプは全て派生プロキシである
2. **レンダリングとシミュレーションの責務分離**: 「何が存在するか」はボリューム場が決定し、「どう見せるか」はレンダラーが担う
3. **ID ベースの追跡**: 全てのオブジェクトは生成から破壊まで一意に追跡可能。破壊後の lineage も保持
4. **version parity**: render proxy と physics proxy は同一の `topology_version` を参照する。不整合は許容しない
5. **将来 GPU スケールへの自然対応**: 品質向上はサンプル予算・輸送精度の増加で達成し、データ表現の作り直しを避ける

### 構造不変条件（Structural Invariants）

以下の不変条件は、全てのトポロジー変更操作（破壊、編集、CSG）の前後で検証する:

- parent/child リンケージの妥当性
- occupancy mask と実割当の一致
- chunk seam coherence（チャンク境界の整合性）
- render/physics の version parity
- per-voxel フィールドの型整合性

### 不採用とした代替アーキテクチャ

以下のアプローチは検討の上、不採用とした。理由を明記しておくことで、将来の再検討時に同じ議論を繰り返すことを防ぐ:

| 代替案 | 不採用理由 |
|--------|-----------|
| **NeRF / Gaussian Splatting を権威状態にする** | 見た目の再現は強いが、破壊トポロジー変更・内部物性・CSG 編集との相性が根本的に悪い。SDF + マテリアル場の方が物理シミュレーションと直結する |
| **純メッシュ + テクスチャ焼き込み** | 初期の見た目は良いが、破壊断面・内部構造の一貫性で急速に破綻する。「割った時に中が正しい」という要件を構造的に満たせない |
| **全面フル Path Tracing のみ** | 品質は最高だが、編集レスポンスや ID 処理の実装効率で不利。ハイブリッドの方がインタラクティブ性と品質を両立できる |

## 決定1: 権威的データ構造

### 背景

石を「体積」として表現し、SDF + マテリアルフィールドを格納するスパースボリュメトリック構造が必要である。この構造は以下の操作に耐える必要がある:

- **読み取り（高頻度）**: レンダリング時の SDF 評価、マテリアル参照、衝突判定
- **値の変更（中頻度）**: 風化度の更新、damage_d の蓄積、moisture の変動
- **トポロジー変更（低頻度・高インパクト）**: 破壊による形状変化、SDF の分割・CSG 演算

### 候補技術: スパースボリューム構造

#### 事前整理: 比較から除外した技術

| 技術 | 除外理由 |
|------|---------|
| Sparse Voxel Octree (SVO) | LOD との親和性は高いが、ランダムアクセス性能が VDB 系の 64-tree に劣る場合がある。また、per-voxel に多チャネルのマテリアルデータを格納する用途では、VDB 系のキャッシュフレンドリーなレイアウトの方が有利。SVO の階層的メタデータ格納の利点は、VDB のマルチレベルグリッドでも実現可能 |
| Neural Implicit (Neural SDF) | 連続的でコンパクトだが、編集（特にトポロジー変更）が根本的に困難。推論コストもリアルタイム向けには高い。**圧縮/ディテール層としての補助的使用は候補に残す**（後述の二層構造を参照） |

#### 候補A: OpenVDB（編集用）+ NanoVDB（GPU 転送/レンダリング用）

**概要:** CPU 側で OpenVDB を権威的編集ストアとして使用し、変更後に NanoVDB に変換して GPU に転送する。

| 観点 | 評価 |
|------|------|
| トポロジー変更 | OpenVDB は完全に動的。ノードの追加・削除が自由 |
| GPU 性能 | NanoVDB はキャッシュフレンドリー、Vulkan/CUDA/GLSL 全対応 |
| 編集→描画レイテンシ | OpenVDB→NanoVDB 変換が必要。小規模変更でも全グリッド再ビルドになりうる |
| エコシステム | 業界標準。VFX パイプラインとの互換性が高い |
| メモリ | OpenVDB と NanoVDB の二重保持が必要 |
| 成熟度 | 製品レベル（OpenVDB: 10年以上の実績、NanoVDB: NVIDIA 公式） |

**ユースケースの向き不向き:**
- 向き: 破壊が散発的で、編集後にまとまった再ビルド時間を確保できるケース
- 不向き: フレームごとに多数の石が同時に破壊されるような高頻度トポロジー変更

#### 候補B: カスタム Sparse Brick Pool + NanoVDB（レンダリング用）

**概要:** 自前で sparse brick pool（ハッシュベースの固定サイズブロック配列）を実装し、編集の自由度を完全に制御する。レンダリング時は NanoVDB に変換。

| 観点 | 評価 |
|------|------|
| トポロジー変更 | 完全に自由。ブリック単位の追加・削除が O(1) |
| GPU 性能 | 設計次第。ブリックプールを GPU バッファに直接マップすれば NanoVDB 変換不要にもできる |
| 編集→描画レイテンシ | 最小化可能（変更されたブリックだけ差分転送） |
| エコシステム | ゼロ。全て自前実装 |
| メモリ | 単一表現で済む可能性がある（GPU 直接マップの場合） |
| 成熟度 | 研究/カスタム。64-tree 等の事例はあるが標準実装はない |

**ユースケースの向き不向き:**
- 向き: 高頻度トポロジー変更、GPU 直接編集、独自最適化が必要なケース
- 不向き: 開発工数が大きい
- VFX 互換性: VDB ↔ カスタム形式の変換レイヤーを用意すれば緩和可能。変換はインポート/エクスポート時のみで、ランタイム性能には影響しない。本プロジェクトはプロシージャル生成が主体のため、外部ツール依存度は低い

#### 候補B 深掘り: GPU 直接マップの実現可能性

候補B の最大のポテンシャルは「NanoVDB を介さず、ブリックプールを GPU バッファに直接マップする」構成にある。これが実現すれば、CPU→GPU 変換のボトルネックが完全に消滅し、候補A/D に対して決定的な性能優位を持つ。ただし実現には複数の技術的課題がある。

**アーキテクチャ概念:**

```
[CPU 側]                              [GPU 側]
Brick Pool (ハッシュテーブル)     ←同一メモリ→    Brick Pool (SSBO/バッファ)
├── Brick[hash(x,y,z)] = データ   Vulkan で      ├── シェーダーから直接参照
├── 変更時: 該当ブリックのみ更新   共有 or        ├── SDF 評価、マテリアル参照
└── 追加/削除: ハッシュ再挿入      差分転送       └── レイマーチング、衝突判定
```

**実現に必要な要素と課題:**

| 要素 | 必要なこと | 難易度 | 備考 |
|------|-----------|--------|------|
| GPU 側トラバーサル構造 | ハッシュテーブル or 階層インデックスをシェーダーで効率的に引ける設計 | 中〜高 | NanoVDB が提供する最適化済みツリートラバーサルを自前で再実装する必要がある |
| キャッシュ効率 | ブリックのメモリレイアウトが GPU のキャッシュライン（128B）に整列 | 中 | NanoVDB はこの点で高度に最適化されている。同等の効率を達成するには慎重なレイアウト設計が必要 |
| SDF 評価の実装 | ブリック境界をまたぐ SDF の補間、勾配計算をシェーダーで実装 | 中 | NanoVDB の SDF サンプラーに相当する機能を自前で書く |
| 差分転送 | 変更されたブリックだけを GPU に送る仕組み（dirty flag + ストリーミング転送） | 低〜中 | Vulkan の sparse buffer binding や staging buffer で実現可能 |
| マルチチャネル対応 | SDF + 6〜10 のマテリアルチャネルを同一ブリック内に格納 or チャネル別ブリックプール | 中 | インターリーブ（1ブリックに全チャネル）vs プレーナー（チャネル別プール）の設計判断が必要 |
| レイマーチング性能 | GPU 上でのスパースボリュームのレイマーチングが NanoVDB のHDDA に匹敵する性能 | 高 | NanoVDB/fVDB の HDDA（Hierarchical Digital Differential Analyzer）は高度に最適化されたアルゴリズム。同等の性能を達成するのが最大の技術的挑戦 |

**先行事例:**

- **64-tree（dubiousconst282, 2024）**: カスタムスパースツリーで SVO 比 3x のメモリ効率と 60% 高速なレイトラバーサルを実証。ただしボクセルレンダリング特化で、SDF + マテリアル場の用途とは異なる
- **Avoyd（enkisoftware, 2024）**: GPU ボクセルレンダリングで Vulkan が OpenGL 比 10x の性能。SVO-DAG ベースのカスタム構造
- **GVDB Voxels（NVIDIA, 旧世代）**: GPU 上でスパースボクセルを直接操作する研究プロジェクト。fVDB の前身的位置づけ

**NanoVDB のレイマーチング性能との比較:**

NanoVDB の HDDA は以下の最適化を含む:
- 階層的な空領域スキップ（空ノードを一括でジャンプ）
- ボクセルレベルでの DDA（最小ステップでボクセル間を移動）
- キャッシュフレンドリーなノードレイアウト（空間的に近いボクセルがメモリ上も近い）

カスタム Brick Pool でこれに匹敵するには:
- ブリック単位の階層インデックス（空ブリックのスキップ）
- ブリック内部での局所 DDA
- Morton/Z-order カーブによるブリック内のボクセル配置

これらは実装可能だが、NanoVDB の 10 年以上の最適化の蓄積に到達するには相当の開発投資が必要。

**判断の軸:**

| 観点 | GPU 直接マップあり | GPU 直接マップなし（NanoVDB 変換） |
|------|-------------------|----------------------------------|
| 編集→描画レイテンシ | ~0ms（差分転送のみ） | 3-30ms（NanoVDB 再ビルド） |
| レイマーチング/SDF 評価性能 | 自前実装の品質次第（NanoVDB より劣る可能性） | NanoVDB の最適化済み性能 |
| 開発コスト | 高（GPU トラバーサル + SDF 評価 + キャッシュ最適化を全て自前） | 中（ブリックプール + NanoVDB 変換のみ） |
| 将来の最適化余地 | 無限（全てが自前なので任意に最適化可能） | NanoVDB の進化に依存 |

**段階的アプローチの可能性:**

GPU 直接マップは「最初から完全実装」ではなく、段階的に導入できる:

1. **Phase 1**: カスタム Brick Pool（CPU）+ NanoVDB 変換（GPU レンダリング用） — 候補B の基本形
2. **Phase 2**: 差分転送の最適化（変更ブリックのみ NanoVDB に反映）
3. **Phase 3**: GPU 上のブリックプール直接参照を一部のパス（衝突判定等）に導入
4. **Phase 4**: レイマーチングを含む完全な GPU 直接マップ（NanoVDB 依存を完全排除）

この段階的アプローチなら、Phase 1 の時点で候補A と同等のエコシステム恩恵を受けつつ、Phase 3-4 で候補B 固有の性能優位を獲得できる。

#### 候補C: fVDB（NVIDIA 2024, GPU ネイティブ）

**概要:** NVIDIA が 2024 年に発表した GPU ネイティブの VDB フレームワーク。NanoVDB 上に構築され、PyTorch 統合、動的階層構造、微分可能オペレータを提供する。Apache 2.0 ライセンス。

| 観点 | 評価 |
|------|------|
| トポロジー変更 | GPU 上で新規グリッド生成が可能。レベルセット追跡、グリッドマージ、適応的細分化をサポート |
| GPU 性能 | テンソルコアによるスパース畳み込み、HDDA による高速レイトレーシング |
| 編集→描画レイテンシ | GPU 上で完結するため CPU-GPU 転送が不要。ただしインプレース変更ではなく新規グリッド生成 |
| エコシステム | OpenVDB プロジェクトの一部。PyTorch 統合あり。ただし主に深層学習ワークフロー向け |
| メモリ | GPU メモリ内で完結 |
| 成熟度 | 研究段階（2024年発表）。主に ML/3D 再構成向けで、ゲームエンジンでの使用実績は未知 |

**ユースケースの向き不向き:**
- 向き: GPU 上で全処理を完結させたいケース。将来的に ML ベースの生成・推論を統合する場合
- 不向き: 現時点でゲームエンジンの本番利用には成熟度が不足。API が ML ワークフロー寄り

#### 候補D: NanoVDB 単体（再ビルド戦略）

**概要:** NanoVDB のみを使用。値変更はインプレース、トポロジー変更時は GPU 上でグリッド再生成ツールを使用する。OpenVDB 13.0（2025年11月）以降、GPU 加速のグリッド生成ツールが拡充されている。

| 観点 | 評価 |
|------|------|
| トポロジー変更 | インプレース不可だが、GPU 加速の再生成ツールで新規グリッドを高速作成 |
| GPU 性能 | 最適。単一表現で CPU-GPU 間の変換が不要 |
| 編集→描画レイテンシ | 値変更は即時。トポロジー変更は再生成が必要だがGPU上で高速 |
| エコシステム | NVIDIA 公式。OpenVDB との相互運用性あり |
| メモリ | 最小。単一表現 |
| 成熟度 | 製品レベル（値変更）+ 発展中（トポロジー再生成ツール） |

**ユースケースの向き不向き:**
- 向き: メモリ効率と GPU 性能を最優先するケース。破壊イベントが十分に局所的な場合
- 不向き: 大規模なトポロジー変更が頻発する場合、再生成コストが蓄積する

### ハイブリッド構成の可能性

破壊イベントは低頻度だが高インパクトであるため、通常時と破壊時で異なる戦略を取る構成が現実的:

| 状態 | データ構造 | 理由 |
|------|-----------|------|
| 通常時（レンダリング、衝突判定、風化更新） | NanoVDB on GPU | 読み取り・値変更に最適化 |
| 破壊イベント発生時 | OpenVDB で SDF 編集 → NanoVDB 再ビルド | トポロジー変更の自由度が必要 |
| 将来（2030+ GPU）| fVDB or GPU ネイティブ再生成 | GPU 上で完結、レイテンシ最小化 |

### 決定 1a: スパースボリューム構造

> **決定: 候補B — カスタム Sparse Brick Pool（4フェーズ段階的実装）**

#### 決定理由

1. **編集→描画レイテンシの最小化**: 変更されたブリックのみの差分転送により、破壊直後のレンダリング応答が最速
2. **適応的細分化との自然な統合**: ブリック単位の O(1) 追加/削除が、マルチ解像度の細分化/粗化操作と直結する（→ 決定 1b への影響大）
3. **GPU 直接マップの可能性**: フェーズを段階的に進めることで、最終的に NanoVDB を介さない GPU バッファ直接参照を実現し、変換ボトルネックを完全排除できる
4. **完全な設計自由度**: per-voxel マテリアル場のレイアウト、キャッシュ最適化、チャネル構成を本プロジェクトの要件に完全特化可能
5. **リスク緩和**: Phase 1 の時点で NanoVDB をレンダリングに利用するため、初期段階から描画品質を確保しつつ段階的にカスタム化を進められる

#### 4フェーズ実装ロードマップ

```
Phase 1: カスタム Brick Pool（CPU）+ NanoVDB 変換（GPU レンダリング用）
├── CPU 側にハッシュベースのスパースブリックプールを実装
├── SDF + マテリアルフィールドをブリック単位で管理
├── 変更後に NanoVDB グリッドへ変換し GPU に転送
└── ✅ この時点で候補A と同等の描画性能 + エコシステム恩恵

Phase 2: 差分転送の最適化
├── dirty flag によるブリック単位の変更追跡
├── 変更ブリックのみを NanoVDB に反映（全グリッド再ビルド回避）
├── Vulkan staging buffer による非同期ストリーミング転送
└── ✅ 編集→描画レイテンシの大幅短縮

Phase 3: GPU 上のブリックプール直接参照（部分導入）
├── 衝突判定、風化シミュレーション等の一部パスで GPU バッファ直接参照
├── ブリックプールを SSBO として GPU に公開
├── SDF 評価・勾配計算をシェーダーで実装
└── ✅ NanoVDB 変換のクリティカルパス依存を段階的に削減

Phase 4: 完全 GPU 直接マップ（NanoVDB 依存排除）
├── レイマーチング、全レンダリングパスで GPU ブリックプール直接参照
├── 階層インデックス + 空領域スキップの自前 HDDA 実装
├── Morton/Z-order によるキャッシュ最適化
└── ✅ 変換ボトルネック完全排除。全処理が GPU 上で完結
```

#### 受容したリスクと緩和策

| リスク | 緩和策 |
|--------|--------|
| 開発工数が大きい | Phase 1 で早期に動作する状態を確保。Phase 3-4 は漸進的に進行 |
| NanoVDB の HDDA に匹敵する性能の達成が困難 | Phase 4 への移行は性能ベンチマークで判断。達成不可なら Phase 2-3 に留まる選択肢を残す |
| VFX ツールとの互換性なし | インポート/エクスポート時に VDB ↔ Brick Pool 変換レイヤーを用意。本プロジェクトはプロシージャル生成主体のため外部依存度は低い |
| NeuralVDB が直接適用不可 | 独自の圧縮戦略を検討（→ 決定 1c で議論） |

### 決定 1b: 解像度戦略 — 二層構造 vs 適応的細分化

#### 背景

「有限の GPU メモリで最大のディテールを表現する」という問題に対して、2つの根本的に異なるアプローチがある。どちらを採用するかは、データの権威性・リアリティの上限・実装アーキテクチャに直結する。

#### アプローチ1: 二層構造（Hyperreal レポートの提案）

```
Layer 1: 権威的構造層（粗い解像度、例: 1mm/voxel）
├── SDF + per-voxel マテリアル記述子を格納
├── 編集・破壊操作の対象
└── 全ての状態変更はここに対して行う

Layer 2: ディテール層（レンダリング時にのみ評価）
├── Layer 1 のマテリアル記述子を条件入力として
├── サブボクセル解像度のディテールをプロシージャル or ML で生成
└── 権威的状態には含まない — キャッシュ可能だが、Layer 1 から常に再生成可能
```

**動作原理**: 権威的データは常に粗い解像度に固定。見た目のディテール（微細な凹凸、SVBRDF の空間変動）はプロシージャルアルゴリズムや拡散モデルが Layer 1 の情報を種にして毎回「導出」する。

#### アプローチ2: 適応的細分化（マルチ解像度ブリック）

```
単一グリッド（可変解像度）
├── カメラ近傍の表面: 0.1mm/voxel（最高解像度）← 必要に応じて細分化
├── 中距離の表面: 1mm/voxel
├── 遠距離 / スリープ中: 5mm/voxel ← 必要に応じて粗化
├── 内部の均質領域: 10mm/voxel
└── 全てが同一の権威的データ形式。解像度だけが異なる
```

**動作原理**: 権威的データそのものの解像度を局所的に変える。カメラが近づけば細分化し、離れれば粗化する。高解像度にしたボクセルには、プロシージャル生成 or ML 推論で権威的データを充填し、そのまま格納する。

#### 両者の本質的な違い

**データの権威性が根本的に異なる:**

| | 二層構造 | 適応的細分化 |
|---|---|---|
| **権威的データの解像度** | 常に粗い（1mm 固定） | 局所的に可変（0.1mm〜10mm） |
| **見た目のディテールは** | 毎回「導出」される（プロシージャル/ML） | 「格納された真実」として存在する |
| **破壊断面の微細構造** | Layer 2 が補完する（プロシージャル品質に依存） | 高解像度の権威的データがそのまま断面に現れる |
| **「それっぽい」リスク** | **あり** — Layer 2 の生成が物理的に正しい保証がない | **低い** — 正しく生成された高解像度データは権威的に永続化される |

最後の行が決定的に重要。プロシージャル生成や拡散モデルは「なぜそうなるのかを理解していない」ため、Layer 2 のディテールは「それっぽい」で止まるリスクがある。適応的細分化では、一度正しく生成したデータが権威的に格納されるため、この問題を構造的に回避できる。

#### 詳細比較

| 観点 | 二層構造 | 適応的細分化 |
|------|---------|------------|
| **メモリ効率** | 予測可能で安定（Layer 1 は固定解像度） | カメラ位置・破壊状況で変動（最悪ケースの見積もりが必要） |
| **設計原則「単一権威状態」** | やや弱い（見た目を支配する Layer 2 が権威外） | **強い**（全データが単一の権威的グリッド） |
| **破壊時の断面品質** | Layer 1 の粗い断面 + Layer 2 のプロシージャル補完 | 高解像度化された領域なら、高精度の断面がそのまま得られる |
| **編集速度** | 粗い Layer 1 のみ変更 → 高速 | 高解像度領域の変更は重い。ただし粗い領域は軽い |
| **LOD の統合** | レンダリング LOD と権威データ LOD が別管理 | **データ構造レベルで LOD が統合** されている |
| **物理シミュレーション精度** | Layer 1 の粗い解像度に制約される | 高解像度領域では物理も高精度に実行可能 |
| **プロシージャル生成への依存** | **高い** — 見た目の品質がプロシージャル品質に直結 | **低い** — 生成は初回のみ、以降は格納データが権威 |
| **実装の複雑性** | 2つのシステム（ボリュームストレージ + 生成エンジン） | 単一システム内のマルチ解像度管理 |

#### 高解像度データの「生成元」問題

適応的細分化は「どの解像度で格納するか」の問題は解決するが、「高解像度データをどう作るか」は別途必要:

- **プロシージャル生成**: 粗いボクセルの mineral_id、porosity 等を種に、細分化時に詳細な鉱物分布・微細構造を生成
- **ML 推論**: 粗い権威データを入力として、高解像度データを推論（超解像の 3D 版）
- **事前計算**: オフラインで高解像度データを生成し、ストリーミングでロード

重要なのは、二層構造でもこれらの生成手段は必要である点。違いは:

- **二層構造**: 毎フレーム（or キャッシュ有効期間中）生成し続ける → 計算コストが**ランタイムに乗る**
- **適応的細分化**: 細分化時に一度だけ生成し格納する → 計算コストは**細分化イベント時のみ**。以降はただの読み取り

#### 候補B（カスタム Brick Pool）との関係

候補B を採用する場合、適応的細分化はブリックプールの設計に自然に組み込める:

```
Brick Pool（マルチ解像度）
├── Level 0: 8mm ブリック — 遠距離 / 内部 / スリープ中
├── Level 1: 2mm ブリック — 中距離
├── Level 2: 0.5mm ブリック — 近距離表面
└── Level 3: 0.1mm ブリック — 破壊領域 / クローズアップ

細分化: 粗いブリックを分割し、子ブリックに高解像度データを充填
粗化: 使われなくなった細かいブリックを統合し、メモリを回収
```

候補B の O(1) ブリック追加/削除は、細分化/粗化操作と相性が良い。ブリック単位で解像度レベルを管理でき、変更は差分転送で GPU に反映可能。

#### 推奨案

**適応的細分化を基本アーキテクチャとして採用。**

シェーダーレベルのマイクロディテール（最終的な法線擾乱、サブピクセルの表面粗さ）のみ、シェーダー内でプロシージャル評価する。これは「二層構造」ではなく、通常のシェーダーパラメータ導出であり、独立したアーキテクチャ層として扱う必要はない。

```
適応的細分化グリッド（権威的、0.1mm〜10mm 可変）
└── シェーダーのマイクロディテール（プロシージャル、法線擾乱程度のみ）
    ※ 独立した「Layer」ではなく、マテリアルシェーダーの通常処理
```

理由:
1. 「単一権威状態」の設計原則に最も忠実
2. ハイパーリアリティの上限がプロシージャル生成品質に制約されない
3. 破壊断面の品質が権威的データから直接得られる
4. 候補B のブリックプール設計と自然に統合可能
5. 「それっぽい」で止まるリスクを構造的に最小化

### 決定 1b

> **決定: 適応的細分化（マルチ解像度ブリック）を基本アーキテクチャとして採用**

#### 決定理由

1. **単一権威状態の原則に最も忠実**: 全データが単一の権威的グリッドに存在し、「見た目を支配するが権威外」という矛盾が生じない
2. **破壊断面の品質が構造的に保証**: 破壊前に対象領域を高解像度化すれば、断面の鉱物分布・微細構造が権威的データとして直接現れる。プロシージャル補完への依存がゼロ
3. **「それっぽい」リスクの構造的排除**: 一度正しく生成した高解像度データを権威的に永続化するため、プロシージャル/ML の品質に上限が制約されない
4. **候補B（カスタム Brick Pool）との自然な統合**: ブリック = 解像度レベルの単位。O(1) のブリック追加/削除が細分化/粗化操作に直結
5. **物理シミュレーション精度**: 高解像度領域では力学シミュレーションも高精度に実行可能（二層構造では粗い Layer 1 の解像度に制約される）

#### メモリ管理戦略

- メモリバジェット上限を設定し、LRU 戦略で制御（最も遠い/古い高解像度ブリックから粗化）
- 2030+ GPU の 48GB+ VRAM を前提とすれば、数百個の石を同時に高解像度で保持可能
- 高解像度データの生成コストは細分化イベント時のみ。ランタイムの毎フレームコストには乗らない

#### シェーダーレベルのマイクロディテール

サブピクセルの法線擾乱・表面粗さのみ、シェーダー内でプロシージャル評価する。これは独立したアーキテクチャ層ではなく、マテリアルシェーダーの通常処理として扱う

#### 受容したリスクと緩和策

| リスク | 緩和策 |
|--------|--------|
| メモリ消費の予測困難性 | LRU ベースのメモリバジェット管理。上限超過時は遠距離ブリックから粗化 |
| 細分化時の高解像度データ生成コスト | 非同期生成（バックグラウンドスレッド）+ 生成完了まで粗い解像度で暫定表示 |
| 細分化/粗化の遷移時のポッピング | 時間的ブレンディングによる滑らかな LOD 遷移 |

### 決定 1c: ブリック単位の圧縮戦略

#### 背景

決定 1a でカスタム Sparse Brick Pool を、決定 1b で適応的細分化を採用した。これにより:

- **NeuralVDB は直接使用不可**: VDB 形式に依存するため、カスタム Brick Pool には適用できない
- **適応的細分化がメモリ消費を増大させる**: 近距離の石には高解像度ブリックが多数生成される
- **ブリック単位で独立に圧縮/デコードできる方式が必要**: ブリックの追加/削除が O(1) であるという設計を壊さないために、圧縮もブリック粒度で完結する必要がある

per-voxel に 19 bytes（コア層）のマテリアル情報を格納する場合、圧縮なしでは:

| シナリオ | アクティブボクセル数 | 非圧縮メモリ | 備考 |
|---------|-------------------|------------|------|
| 石 1 個（30cm, 1mm/voxel） | 170K | 3.2 MB | ナローバンド ±3 voxel |
| 石 100 個 | 17M | 323 MB | |
| 石 1000 個 | 170M | 3.2 GB | |
| 石 100 個 + 適応的細分化（近距離 10 個が 0.1mm） | ~187M | ~3.6 GB | 近距離の 10 個が 100x 高解像度 |

2030+ GPU の 48GB+ VRAM を前提とすれば、非圧縮でも技術的に収まるが、シーン規模の拡大余地を確保するために圧縮は重要。

#### 圧縮の適用レイヤー

```
[ランタイム GPU メモリ（VRAM）]
├── アクティブブリック: 非圧縮（シェーダーから直接アクセス）
├── 近距離ブリック: 非圧縮
├── 中距離ブリック: 量子化圧縮（GPU 上で即時デコード可能）
└── スリープ中ブリック: ブロック圧縮 or アンロード

[ストリーミング/ストレージ（ディスク / システムメモリ）]
├── 全ブリック: 高圧縮（Neural Brick Codec or ブロック圧縮）
└── ロード時にデコードして VRAM に展開
```

重要な設計原則: **シェーダーがアクセスするブリックは常に非圧縮 or GPU デコード可能な軽量圧縮**。高圧縮はストレージ/ストリーミング層にのみ適用する。

#### 候補技術

##### 候補A: チャネル別量子化（即時導入可能）

各フィールドの値域と必要精度に応じて、格納ビット幅を最適化する。

| チャネル | 元の型 | 量子化後 | 圧縮率 | 精度への影響 |
|---------|--------|---------|--------|------------|
| SDF | float16 (2B) | int8 (1B) + スケール | 2x | ±127 段階。ナローバンド幅 6 voxel なら十分 |
| mineral_id | uint8 (1B) | uint8 (1B) | 1x | 離散値。圧縮不可 |
| volume_fraction[4] | float16×4 (8B) | uint8×4 (4B) | 2x | 256 段階の混合比。視覚的に十分 |
| porosity | float16 (2B) | uint8 (1B) | 2x | 0-1 を 256 段階 |
| grain_size | float16 (2B) | uint8 (1B) | 2x | 相対スケール |
| weathering_degree | float16 (2B) | uint8 (1B) | 2x | 0-1 を 256 段階 |
| microcrack_density | float16 (2B) | uint8 (1B) | 2x | 0-1 を 256 段階 |
| damage_d | float16 (2B) | uint8 (1B) | 2x | 0-1 を 256 段階 |

**量子化後のコア層**: 1 + 4 + 1 + 1 + 1 + 1 + 1 + 1 = **11 bytes/voxel**（非圧縮 19B から 42% 削減）

SDF は別格の扱い: ブリック全体のスケール（float32）+ オフセット（float32）をヘッダーに格納し、ボクセルは int8 の相対値のみ。これにより高精度を維持しつつ圧縮。

| 長所 | 短所 |
|------|------|
| 実装が単純。GPU でのデコードコストがほぼゼロ | 圧縮率は 2x 程度が限界 |
| ブリック単位で完全に独立 | 空間相関を活用しない |
| Phase 1 から即座に導入可能 | |

##### 候補B: ブリック内ブロック圧縮（BC7/ASTC 応用）

GPU テクスチャ圧縮の技術をボリュームデータに応用する。ブリック内の 4×4×4 ボクセルブロックを固定サイズに圧縮。

| 技術 | 圧縮率 | GPU デコード | 適用対象 |
|------|--------|------------|---------|
| BC7 | 4x〜6x | ハードウェアデコード | 連続値チャネル（SDF, porosity 等） |
| ASTC | 可変（4x〜36x） | ハードウェアデコード | 連続値チャネル。圧縮率と品質のトレードオフ調整可 |
| カスタムブロック圧縮 | 3x〜8x | シェーダーデコード | 全チャネル対応可能 |

**実装方法**: ブリック（例: 8^3 ボクセル）を 3D テクスチャとしてパックし、GPU のテクスチャ圧縮ハードウェアを活用。

| 長所 | 短所 |
|------|------|
| GPU ハードウェアデコードで追加コストなし | 3D テクスチャとして格納する必要がある（SSBO 直接アクセスとの両立が課題） |
| 4x〜6x の圧縮率 | 離散値（mineral_id）には適用不可 |
| 業界標準の圧縮品質 | ブロック境界でのアーティファクトの可能性 |

**Phase 3-4 との互換性の課題**: GPU 直接マップ（SSBO）とテクスチャ圧縮は異なるアクセスパスを使う。Phase 1-2（NanoVDB 変換）では問題ないが、Phase 3-4 で SSBO 直接参照に移行する際、圧縮ブリックのアクセス方法を再設計する必要がある。

##### 候補C: Neural Brick Codec（ストレージ/ストリーミング用）

ブリック単位の小型ニューラルネットワークエンコーダ/デコーダ。NeuralVDB の思想をブリック粒度に適用。

**動作原理:**

```
エンコード（オフライン/非同期）:
  Brick(8^3 × N channels) → Encoder → Latent Vector(数十〜数百 bytes)

デコード（ロード時）:
  Latent Vector → Decoder → Brick(8^3 × N channels)
```

| 長所 | 短所 |
|------|------|
| 極めて高い圧縮率（10x〜50x） | デコードに推論コストが必要 |
| 空間相関を学習的に活用 | エンコーダ/デコーダの学習が必要 |
| チャネルの型を問わない（離散・連続の両方に対応可能） | リアルタイムのランダムアクセスデコードは困難 |

**適用範囲**: ディスク → VRAM のストリーミング転送時にのみ使用。VRAM 上のアクティブブリックは非圧縮で格納。

| 長所 | 短所 |
|------|------|
| ディスク使用量を劇的に削減（ワールドの永続化） | ランタイムの VRAM 圧縮には使えない |
| NeuralVDB と同等のコンセプトをブリック粒度で実現 | 学習パイプラインの構築が必要 |
| 将来の ML 統合（プロシージャル生成等）と技術スタックが共通 | Phase 1 での優先度は低い |

##### 候補D: スパース性の活用（ゼロコスト圧縮）

適応的細分化の設計そのものが最大の「圧縮」である。

- **粗い解像度のブリック**: 遠距離の石は 5mm〜10mm/voxel。同じ石でもボクセル数が 1/1000 になる
- **内部の省略**: ナローバンド SDF により、表面から離れた内部ボクセルは存在しない
- **空ブリックの非存在**: ハッシュテーブルに空ブリックは登録されない → メモリ消費ゼロ
- **均質領域の粗化**: 内部が均質な鉱物の場合、低解像度ブリックのまま保持

これは「圧縮技術」というより「データ構造の設計による本質的なスパース性」だが、メモリ効率への貢献は他のどの圧縮技術よりも大きい。

#### チャネル別の最適圧縮マッピング

| チャネル | 型 | 特性 | ランタイム（VRAM） | ストレージ（ディスク） |
|---------|------|------|-----------------|-------------------|
| SDF | float16 | 連続、高空間相関 | int8 + ブリックヘッダー（スケール/オフセット） | Neural Brick Codec |
| mineral_id | uint8 | 離散 | 非圧縮 | パレットインデックス + RLE |
| volume_fraction[4] | float16×4 | 連続、スパース（多くが [1,0,0,0]） | uint8×4 | Neural Brick Codec |
| porosity | float16 | 連続 | uint8 | Neural Brick Codec |
| grain_size | float16 | 連続 | uint8 | Neural Brick Codec |
| weathering_degree | float16 | 連続 | uint8 | Neural Brick Codec |
| microcrack_density | float16 | 連続、多くが 0 | uint8 | RLE + 量子化 |
| damage_d | float16 | 連続、多くが 0 | uint8 | RLE + 量子化 |

#### 段階的導入ロードマップ

| フェーズ | 圧縮戦略 | 圧縮率（ランタイム） | 圧縮率（ストレージ） |
|---------|---------|-------------------|-------------------|
| Phase 1 | 量子化のみ（候補A） | ~1.7x（19B→11B/voxel） | 量子化 + zstd |
| Phase 2 | 量子化 + ブロック圧縮の連続チャネル適用 | ~2.5x〜3x | 量子化 + zstd |
| Phase 3+ | 量子化 + Neural Brick Codec（ストレージ） | ~2.5x〜3x | 10x〜50x |

#### スケーラビリティ再見積もり（量子化後）

| シナリオ | ボクセル数 | 非圧縮 (19B) | 量子化後 (11B) | 備考 |
|---------|----------|------------|--------------|------|
| 石 1 個 | 170K | 3.2 MB | 1.9 MB | |
| 石 100 個 | 17M | 323 MB | 187 MB | |
| 石 1000 個 | 170M | 3.2 GB | 1.9 GB | |
| 石 100 個 + 適応的細分化 | ~187M | ~3.6 GB | ~2.1 GB | 近距離 10 個が高解像度 |

48GB VRAM を前提とすれば、量子化だけでも十分なヘッドルームがある。ブロック圧縮を追加すれば さらに余裕が生まれる。

#### 推奨案

**段階的に圧縮を強化する三層戦略:**

1. **即時（Phase 1）**: チャネル別量子化（19B→11B/voxel）。実装コストが低く効果が大きい
2. **中期（Phase 2-3）**: 連続チャネルへのブロック圧縮適用。GPU テクスチャ圧縮ハードウェアの活用を検討
3. **長期（Phase 3+）**: Neural Brick Codec によるストレージ圧縮。ディスク/ネットワーク経由のストリーミングに適用

最も重要なのは、**適応的細分化そのもの（候補D）が最大の圧縮**であるという認識。遠距離の石を粗い解像度で保持するだけで、ボクセル数が桁違いに削減される。上記の量子化・ブロック圧縮はその上に乗る追加の最適化。

## 決定2: Per-Voxel マテリアルフィールド設計

### 背景

各アクティブボクセルにどのようなデータを格納するかは、外観・物理・破壊の全サブシステムの能力を規定する基盤的決定である。ストレージコストとシミュレーション精度のバランスが核心的トレードオフ。

### 候補設計

#### 候補A: Hyperreal 案（コンパクト + ランタイム導出）

```
mineral_id:          uint8     (1 byte)
crystal_orientation: float16×3 (6 bytes)
porosity:            float16   (2 bytes)
microcrack_density:  float16   (2 bytes)
weathering_degree:   float16   (2 bytes)
grain_size:          float16   (2 bytes)
─────────────────────────────────
合計: 15 bytes/voxel
```

力学パラメータ（E, nu, Gc, sigma_t）は mineral_id と porosity からルックアップテーブルで導出。

| 長所 | 短所 |
|------|------|
| メモリ効率が高い | 風化や損傷で鉱物学的ルックアップから乖離する非標準状態を表現できない |
| 導出ロジックが一箇所に集約 | mineral_id が離散的（uint8）なため、粒界遷移帯の混合表現に限界 |
| フィールド数が少なく管理が容易 | 力学パラメータの局所的オーバーライドが不可能 |

#### 候補B: Structural 案（明示的力学場 + 混合対応）

```
mineral_id:            uint8       (1 byte)
volume_fraction[4]:    float16×4   (8 bytes)  ← 最大4鉱物の混合比
grain_size:            float16     (2 bytes)
porosity:              float16     (2 bytes)
layering_orientation:  float16×3   (6 bytes)
E:                     float16     (2 bytes)  ← ヤング率
nu:                    float16     (2 bytes)  ← ポアソン比
Gc:                    float16     (2 bytes)  ← 破壊靭性
sigma_t:               float16     (2 bytes)  ← 引張強度
anisotropy_tensor:     float16×3   (6 bytes)
damage_d:              float16     (2 bytes)
moisture:              float16     (2 bytes)
temperature:           float16     (2 bytes)
─────────────────────────────────
合計: 39 bytes/voxel
```

| 長所 | 短所 |
|------|------|
| 風化・損傷による非標準的な力学状態を正確に表現 | メモリ使用量が候補Aの約2.6倍 |
| volume_fraction で粒界遷移帯の連続的混合が可能 | フィールド数が多く、一貫性維持の管理コストが増大 |
| damage_d が永続的に蓄積（セーブ/ロードで完全復元） | 初期化時に全フィールドの整合的な値設定が必要 |
| moisture/temperature により風化の動的連成が可能 | 圧縮率が悪化する可能性（NeuralVDB の効果が減少しうる） |

#### 候補C: 統合案（二層フィールド）

```
[永続的コア層 — 全ボクセルに格納]
mineral_id:          uint8       (1 byte)
volume_fraction[4]:  float16×4   (8 bytes)
porosity:            float16     (2 bytes)
grain_size:          float16     (2 bytes)
weathering_degree:   float16     (2 bytes)
microcrack_density:  float16     (2 bytes)   ← 破壊閾値・表面粗さに影響
damage_d:            float16     (2 bytes)
─────────────────────────────────
コア: 19 bytes/voxel

[力学拡張層 — アクティブシミュレーション領域のみに格納]
E:                   float16     (2 bytes)
nu:                  float16     (2 bytes)
Gc:                  float16     (2 bytes)
sigma_t:             float16     (2 bytes)
anisotropy_tensor:   float16×3   (6 bytes)
moisture:            float16     (2 bytes)
temperature:         float16     (2 bytes)
─────────────────────────────────
拡張: 18 bytes/voxel（必要な領域のみ）

[Entity レベルのメタデータ（per-voxel ではなく per-石）]
topology_version:    uint32      (4 bytes)   ← render/physics の version parity 用
```

**topology_version について**: これは per-voxel ではなく per-Entity（per-石）のメタデータ。トポロジー変更操作（破壊、CSG）が完了するたびにインクリメントされ、render proxy と physics proxy が同じバージョンを参照していることを保証する。決定0の構造不変条件「version parity」の実装基盤。

| 長所 | 短所 |
|------|------|
| 通常時は 19 bytes/voxel でメモリ効率を確保 | 二層管理の複雑性 |
| 破壊/風化シミュレーション対象の領域だけ力学場を展開 | 層間の整合性チェックが必要 |
| damage_d がコア層にあり永続化される | 力学拡張層のライフサイクル管理が追加で必要 |
| volume_fraction で混合対応 | — |
| mineral_id からの初期導出 + damage/weathering による乖離の両方をサポート | — |

### トレードオフの軸

| 軸 | 候補A | 候補B | 候補C |
|----|-------|-------|-------|
| メモリ/voxel | 15B (最小) | 39B (最大) | 19B + 18B (対象領域のみ) |
| 粒界表現 | 離散的 (uint8) | 連続的 (volume_fraction) | 連続的 |
| 非標準力学状態 | 不可 | 完全対応 | 対象領域で対応 |
| 損傷の永続化 | FractureState コンポーネント（曖昧） | damage_d フィールド（明確） | damage_d フィールド（明確） |
| 実装複雑性 | 低 | 中 | 中〜高 |
| NeuralVDB 圧縮効率 | 高（フィールド少） | 低〜中（フィールド多） | 中（コア層は高） |

### crystal_orientation について

候補A には `crystal_orientation: float16×3` が含まれているが、候補B/C では `layering_orientation: float16×3` と `anisotropy_tensor: float16×3` に分離されている。

- `crystal_orientation` は結晶粒レベルの方位（劈開面方向を決定）
- `layering_orientation` は堆積層の方向（縞模様の方向を決定）
- `anisotropy_tensor` は力学的異方性（破壊伝播方向に影響）

これらは異なる物理量であり、統合案ではどう扱うかを決める必要がある:
- **案1**: crystal_orientation をコア層に含め、grain_size と合わせてプロシージャル生成の種とする。layering_orientation は別途 Reaction-Diffusion の入力として外部パラメータ化
- **案2**: anisotropy_tensor をコア層に格納し、crystal_orientation と layering_orientation はここから導出

### 推奨案

**候補C（統合・二層フィールド）を推奨。**

理由:
1. 全ボクセルに力学場を持つのはメモリ過多（大半の石は破壊されない）
2. しかし damage_d と volume_fraction はコア層に必要（損傷蓄積と粒界表現は常時必要）
3. 力学拡張層を「破壊/風化の対象領域のみ」に限定することで、Structural の精度と Hyperreal の効率を両立
4. 初期の力学パラメータは mineral_id + volume_fraction + porosity からルックアップ導出し、風化/損傷で乖離した場合のみ拡張層の明示的値が優先される

## 決定3: 形状表現とメッシュ抽出

### 背景

両レポートとも **SDF（Signed Distance Field）を権威的形状表現**とし、**Dual Contouring でメッシュを抽出**する点で合意している。ここでの決定事項は、SDF のストレージ形式と Dual Contouring の具体的バリアントの選択である。

### SDF ストレージ: 決定1と連動

SDF は決定1で選定したスパースボリュメトリック構造に格納される。追加の論点:

- **SDF 解像度**: 石のサイズに応じた適応的解像度。小石: 0.5mm/voxel、大岩: 2mm/voxel が目安
- **SDF 帯域幅**: 表面から ±3 voxel のナローバンド SDF で十分（内部は符号のみ）
- **SDF 勾配**: 法線計算に使用。格納 vs オンデマンド計算のトレードオフ
  - 格納: 追加 6 bytes/voxel（float16×3）だがアクセスは即時
  - オンデマンド: 近傍 6 点の SDF 値から中心差分で計算。追加ストレージ不要だが計算コスト

### メッシュ抽出 vs 直接レンダリング

2030+ GPU の計算バジェットを前提とすると、SDF をメッシュに変換せず**直接レイマーチング**でレンダリングする選択肢も検討に値する:

| アプローチ | 長所 | 短所 |
|-----------|------|------|
| **メッシュ抽出（DC）→ ラスタライズ** | GPU のラスタライズパイプラインをフル活用。既存のレンダリング技法（シャドウマップ、G-Buffer 等）と自然に統合 | メッシュ抽出コスト。トポロジー変更のたびに再メッシュが必要。メッシュは近似であり SDF の精度を失う |
| **SDF 直接レイマーチング** | メッシュ抽出不要。SDF の精度を完全に保持。破壊後の即時レンダリングが可能 | レイマーチングのコストが高い（特に大量の石）。既存のラスタライズベースのパイプラインとの統合が複雑 |
| **ハイブリッド: ラスタライズ + 近距離レイマーチング** | 遠方はメッシュで効率的に、近距離は SDF で高精度に | 2つのパスの管理が必要。LOD 遷移の設計が複雑 |

**推奨**: メッシュ抽出（DC）を基本とする。理由は、GPU 駆動レンダリング（Visibility Buffer、メッシュレット）のパイプラインがメッシュベースの入力を前提としており、レンダリングパイプライン全体（第4章で議論）との統合が自然であるため。ただし、破壊直後の**暫定レンダリング**として SDF 直接レイマーチングを使用する余地は設計に組み込む。

### Dual Contouring バリアント

| バリアント | 概要 | 長所 | 短所 | 成熟度 |
|-----------|------|------|------|--------|
| Classic DC | Hermite データ（交差点 + 法線）から QEF 最小化でメッシュ頂点を配置 | シャープエッジ保存、理論が明快 | 自己交差の可能性、QEF が ill-conditioned になりうる | 確立済み |
| Neural DC (NDC, 2022) | ニューラルネットワークで QEF 解を予測 | 自己交差のない高品質メッシュ | 推論コスト、学習データが必要 | 研究段階 |
| Self-Supervised DC (SDC, CVPR 2024) | 教師データなしで学習。SDF のみから高品質メッシュを生成 | 教師データ不要、Classic DC を品質で上回る | 推論コスト | 研究段階 |
| Occupancy-Based DC (ODC, 2024) | SDF が不完全な領域でも動作（占有率ベース） | 破壊直後の不完全 SDF でも動作可能 | 精度が完全 SDF 入力に劣る場合あり | 研究段階 |
| FlexiCubes (2023) | 微分可能メッシュ抽出。重み付きデュアルグリッド | 学習パイプラインとの統合が容易 | ゲームエンジンでの実用性は未検証 | 研究段階 |

### ユースケース別の使い分け

| ユースケース | 推奨バリアント | 理由 |
|-------------|--------------|------|
| 通常のメッシュ抽出（初回ロード、破壊後の再メッシュ） | Classic DC | 高速、決定論的、十分な品質 |
| 高品質モード / クローズアップ | SDC or NDC | シャープエッジの品質向上、自己交差の回避 |
| 破壊直後の暫定メッシュ | ODC | SDF が完全に再計算される前でも暫定メッシュを提供 |
| ML ベースの形状生成パイプライン | FlexiCubes | 微分可能性が必要な場合 |

### 推奨案

**Classic DC を基本実装とし、SDC/ODC を将来の品質向上オプションとして設計に組み込む。**

理由:
1. Classic DC は実績があり、シャープエッジ保存が石の劈開面表現に不可欠
2. 破壊イベント後の高速再メッシュには決定論的で高速な Classic DC が最適
3. SDC は教師データ不要で学習可能なため、将来的な品質向上の導入障壁が低い
4. ODC は破壊直後の暫定メッシュ提供に価値があるが、フォールバック用途に限定
5. メッシュ抽出インターフェースを抽象化しておけば、バリアントの差し替えは容易

### SDF 勾配に関する推奨

**オンデマンド計算を基本とし、衝突判定の高頻度アクセス領域のみキャッシュする。**

理由: 法線はレンダリング時にシェーダーで計算可能（GPU の計算能力を考慮すると、ストレージ削減の方が有利）。ただし SDF-SDF 衝突判定では勾配への高頻度アクセスが発生するため、アクティブな衝突領域にのみ勾配キャッシュを生成する。

## 決定4: ゲームオブジェクトモデル（ECS + ID 体系）

### 背景

石・植生・破壊フラグメントをゲームオブジェクトとして管理し、選択・移動・破壊・親子関係を実現する。ECS（Entity-Component System）の採用は両レポートで合意。決定すべきは ECS の具体的実装と ID 体系の設計。

### 4A: ECS 実装の選定

#### 候補

| 実装 | 言語 | アーキテクチャ | 長所 | 短所 |
|------|------|--------------|------|------|
| **Flecs** | C99 + C++ API | アーキタイプベース | 成熟（v4.x）、豊富なドキュメント、ビルトインの関係性（parent/child）、クエリ言語、モニタリングUI、マルチスレッド対応 | C ベースのため C++ 的な型安全性が間接的 |
| **EnTT** | C++17 ヘッダーオンリー | スパースセットベース | 広く採用（Minecraft Bedrock 等）、単一コンポーネントアクセスが高速、add/remove が高速 | マルチコンポーネントクエリが Flecs に劣る、関係性の組み込みサポートなし |
| **カスタム軽量 ECS** | C++20 | 自由設計 | 本プロジェクトの要件に完全特化可能 | 開発・保守コストが大、バグリスクが高い |

**注記**: Hyperreal レポートで言及された「Astra (C++20, 1.05ns/entity iteration)」は、調査の結果、広く知られたプロジェクトではなく、情報源の信頼性が不明である。候補からは除外し、実績のある Flecs と EnTT を主要候補とする。

#### 本プロジェクトでの要件とのマッチング

| 要件 | Flecs | EnTT | カスタム |
|------|-------|------|---------|
| 石-植生の親子関係 | ビルトインの `ChildOf` 関係 | 手動実装が必要 | 自由 |
| 破壊後のフラグメント分割（1→N Entity 生成） | Entity の動的追加が高速 | Entity 追加は高速だがコンポーネント付与のバッチ処理が Flecs に劣る | 自由 |
| マルチコンポーネントクエリ（VolumetricBody + RigidBody + RenderProxy の同時イテレーション） | アーキタイプベースで高速 | スパースセットベースで若干劣る | 設計次第 |
| デバッグ・モニタリング | Flecs Explorer（WebUI）でリアルタイム監視 | 外部ツールなし | 自前実装が必要 |
| C++20 統合 | C++ API は C++11 互換、C++20 機能は直接活用しない | C++17 ヘッダーオンリー | C++20 フル活用可能 |

### 4B: ID 体系の設計

Structural レポートの ID 体系が、本プロジェクトの要件（選択、破壊後の lineage 追跡、石と植生の識別）に最も直接的に対応している:

| ID | 型 | 用途 | ECS での実装 |
|----|----|------|-------------|
| `instance_id` | uint32 | ゲーム編集用の個体 ID。選択、ハイライト、再配置 | Entity ID そのもの、または専用コンポーネント |
| `fragment_id` | uint32 | 破壊後のフラグメント固有 ID | 新 Entity の ID |
| `parent_fragment_id` | uint32 | 破壊元の Entity ID。lineage 追跡 | `ChildOf` 関係 or 専用コンポーネント |
| `submaterial_id` | uint16 | 岩体内サブ領域識別（石英脈、特定鉱物塊） | VolumetricBody 内のラベルフィールド |
| `bio_patch_id` | uint32 | 苔・植物パッチ単位識別 | 植生子 Entity の ID |

#### lineage 追跡の設計

破壊時に Stone_001 が 3 つのフラグメントに分割される場合:

```
破壊前:
  Entity: Stone_001 (instance_id=1001)

破壊後:
  Entity: Stone_001_frag_0 (instance_id=1001, fragment_id=2001, parent=Stone_001)
  Entity: Stone_001_frag_1 (instance_id=1001, fragment_id=2002, parent=Stone_001)
  Entity: Stone_001_frag_2 (instance_id=1001, fragment_id=2003, parent=Stone_001)
```

- `instance_id` は共通 → 「元は同じ石」の追跡が可能
- `fragment_id` は一意 → 各フラグメントの個別操作が可能
- `parent` 関係 → 再帰的な破壊（フラグメントをさらに割る）にも対応

### 4C: Version Parity の設計

決定0で定義した「render proxy と physics proxy は同一の `topology_version` を参照する」不変条件を、ECS 上でどう実装するか:

**方針**: `VolumetricBody` コンポーネントが `topology_version` を保持する。トポロジー変更操作が完了すると:

1. `topology_version` をインクリメント
2. `RenderProxy` の再生成をキューに追加（非同期）
3. `RigidBody` のコリジョンシェイプ再生成をキューに追加（非同期）
4. 両方の再生成が完了するまで、旧バージョンの proxy を使用し続ける
5. 両方が完了した時点で、同時にアトミックに切り替え

これにより、render と physics が異なるバージョンの形状を参照する不整合（石が「浮く」「めり込む」等の視覚的破綻）を防止する。

### 4D: コンポーネント設計

両レポートの提案を統合したコンポーネント設計案:

| コンポーネント | 内容 | 対応レポート |
|--------------|------|-------------|
| `VolumetricBody` | NanoVDB/OpenVDB ハンドル、SDF + マテリアル場への参照 | 両方 |
| `RigidBody` | 質量、慣性テンソル、反発係数、摩擦係数 | Hyperreal |
| `RenderProxy` | 抽出メッシュ、マテリアルバインド、LOD レベル | 両方 |
| `Interactable` | 選択可能フラグ、ハイライト状態、操作モード | Hyperreal |
| `FractureState` | damage_d サマリー、亀裂ネットワーク、構造健全性指標 | 両方 |
| `Lineage` | instance_id、fragment_id、parent_fragment_id | Structural |
| `CouplingField` | 石-植生間の接触面積、付着強度、水分輸送 | Structural |
| `GrowthState` | 植生の成長パラメータ、根の浸透状態 | Hyperreal |
| `VegetationBody` | L-system 状態、メッシュ参照 | Hyperreal |

### 推奨案

**ECS 実装: Flecs を推奨。ID 体系: Structural の設計を採用。**

理由:
1. **Flecs**: ビルトインの親子関係 (`ChildOf`) が石-植生階層と破壊後の lineage に直結。アーキタイプベースのクエリが VolumetricBody + RigidBody + RenderProxy の同時イテレーションに最適。Flecs Explorer によるデバッグが開発効率を大幅に向上
2. **EnTT を選ばない理由**: 関係性の手動実装コストが高く、マルチコンポーネントクエリ性能が劣る。単一コンポーネントアクセス速度の優位性は本プロジェクトのユースケースでは活きにくい
3. **カスタムを選ばない理由**: ECS は本プロジェクトのコア差別化要因ではない。既存の成熟した実装を使い、開発リソースをレンダリングと物理に集中すべき
4. **ID 体系**: Structural の instance_id / fragment_id / parent_fragment_id / bio_patch_id は、選択・破壊・lineage・植生識別の全要件を最小限の構造で満たす

## 決定間の相互依存マップ

以下のマップは、各決定が他の決定にどう影響するかを示す。矢印は「この決定がこの決定に影響する」を意味する。

```
決定0: 設計原則 ──────────────→ 全ての決定の判断基準
       │
       ▼
決定1a: スパースボリューム構造
       │
       ├──→ 決定1b: 二層構造（Layer 1 のストレージ形式が決定1aに依存）
       │
       ├──→ 決定1c: 圧縮レイヤー（NeuralVDB は VDB 系でのみ利用可能）
       │
       ├──→ 決定2: マテリアルフィールド（ストレージの制約が per-voxel バイト数に影響）
       │
       └──→ 決定3: メッシュ抽出（SDF の格納形式が DC の入力に影響）

決定2: マテリアルフィールド
       │
       ├──→ 決定1c: 圧縮（フィールド数と型が圧縮効率に影響）
       │
       └──→ 第2章以降: テクスチャ技術（マテリアル場のフィールドがテクスチャ生成の入力）
            第3章以降: 破壊シミュレーション（力学場の設計が破壊 solver の能力を規定）

決定4: ECS + ID 体系
       │
       └──→ 第5章以降: 植生統合（親子関係と coupling field の設計）
            第6章以降: ゲームオブジェクト認識（ID 体系が選択・ハイライトの基盤）
```

**推奨される決定順序**: 0 → 1a → 1b → 1c → 2 → 3 → 4

理由: 各決定は前の決定の結果に依存する。特に決定1a（スパースボリューム構造）は決定1c（圧縮）と決定2（フィールド設計）の両方に制約を課すため、最初に確定すべき。
