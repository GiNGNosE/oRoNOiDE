# 決定 3: 形状表現とメッシュ抽出

**議論ファイル**: `engine/docs/discussion/03_shape_representation_discussion.md`（未作成）

## 設計方針

> **SDF直接レイマーチング + SDF直接物理。ランタイムメッシュ生成なし。**

SDFが形状の権威的定義（決定1a）である以上、メッシュは派生データに過ぎない。派生データの生成・同期・二重管理を排除し、全パイプラインがSDFを直接参照する設計を採る。

---

## レンダリング: SDF Sphere Tracing

パストレーサーの全レイ（一次レイ、二次レイ、シャドウレイ）がSBP上のSDFを直接sphere tracingで歩く。

### 2段階トラバーサル

```
Level 1: ブリックレベルDDA
  レイがSBPのブリック構造を通過する順序を決定
  空ブリック（SBPに存在しない）はスキップ
  sdf_min > 0 のブリックは距離に応じてスキップ（後述）

Level 2: ブリック内sphere tracing
  int8 SDFを読みながらレイを進める
  SDF ≈ 0 で表面到達 → シェーディングへ
  ブリック境界到達 → Level 1に戻り次のブリックへ
```

### ブリックスキップ加速

ブリックヘッダーにSDF min/maxを追加:

```
brick_header（既存フィールドに追加）:
  sdf_min: int8   (1B)   -- ブリック内SDFの最小値
  sdf_max: int8   (1B)   -- ブリック内SDFの最大値
```

- `sdf_min > 0`: ブリック全体が表面外部。レイの現在位置からの距離がsdf_minより大きければブリック丸ごとスキップ
- `sdf_max < 0`: ブリック全体が内部。同様にスキップ
- コスト: 2B/ブリックヘッダー。SDF更新時にmin/maxを再計算するだけ

典型的なシーンでは表面近傍のブリックは全体の20-30%。レイが通過するブリックの大半でsphere tracingを完全回避できる。

### 追加加速の候補（プロファイリング後に検討）

| 手法 | 概要 | コスト | 導入判断 |
|------|------|--------|----------|
| 階層的距離場 | ブリックグループ単位の粗いSDF | 低 | LODブリックが同等の役割を果たす可能性。重複を確認してから |
| ブリック内サブブロックmin | 2x2x2サブブロック単位のsdf_min | 4B/ブリック | 効果が限定的な可能性。プロファイリング後 |

## int8量子化とsphere tracingの精度

### 量子化ポリシーによる安全性保証

sphere tracingはSDFの値を「安全に進める距離」として使う。SDFが過大評価されるとレイが表面を突き抜けるリスクがある。

**量子化時の丸め方向を制御することで、過大評価を原理的に排除する:**

```
量子化ルール:
  sdf > 0: floor（切り捨て） → 実際より小さい → レイが短く進む（安全）
  sdf < 0: ceil（切り上げ）  → 絶対値が小さい → 内部判定が保守的（安全）
```

保守的係数（step *= 0.9等）は不要。量子化ポリシーだけで解決する。

### 表面近傍の有効精度

ブリックヘッダーのscale/offset（決定1c）により:

```
実際のSDF = int8_value * scale + offset
有効精度 = (sdf_max - sdf_min) / 255

表面近傍ブリック（値域 -2mm〜+2mm の場合）:
  有効精度 ≈ 4mm / 255 ≈ 0.016mm
  ボクセル解像度 0.25mm に対して十分
```

## 衝突判定: SDF直接クエリ

自前物理エンジンがSDFを直接参照する。メッシュベースの衝突判定は採用しない。

### SDFネイティブ衝突判定の構造的優位性

- **距離クエリ O(1)**: 任意の点から最近接表面までの距離がSDF値の読み取りだけで得られる。メッシュベースではBVHトラバーサルが必要
- **接触法線が即座に得られる**: 格納済み法線チャネル（snorm8x2）を直接参照
- **破壊との自動同期**: SDFが変更されれば衝突形状も即座に更新。メッシュ再生成+BVH再構築が不要

### SDF-SDF衝突（石同士）

一方のSDFの表面点を他方のSDFで評価する方式:

```
1. Entity AのSDF=0近傍ボクセルを列挙（表面点集合）
2. 各表面点をEntity Bのローカル座標に変換
3. Entity BのSDFを評価 → 負なら貫通
4. 貫通深度 = |SDF_B(point)|、法線 = SDF_Bの勾配（格納済み法線）
```

SBPの構造がそのまま表面点の列挙に使えるため、追加のデータ構造は不要。

## Dual Contouring の位置付け

**ランタイムパイプラインの外に置く。オフラインユーティリティとして位置付ける。**

| 用途 | 備考 |
|------|------|
| エクスポート（3Dプリント、.obj/.stl） | ユーザー要求時にのみ実行 |
| デバッグ可視化 | ワイヤーフレーム表示でSDF形状を確認 |
| Mod/UGC | メッシュとして書き出す需要がある場合 |

DCをMCではなく選択する理由: 鉱物の劈開面や結晶形状のシャープエッジを保持するため。格納済み法線（snorm8x2）をDCのHermiteデータとして直接利用可能。

実装優先度は低い。必要になった時点で実装する。

## NanoVDBからの移行パス

| 役割 | Phase 1 | Phase 3+ |
|------|---------|----------|
| 空間加速構造 | NanoVDB tree | SBPブリックハッシュ + DDA |
| レイトラバーサル | HDDA | ブリックレベルDDA + ブリック内sphere tracing |
| SDF格納 | NanoVDB grid（SBPから変換） | SBPブリック直接（int8 + header） |
| 衝突判定 | NanoVDB SDF query | SBP直接query |

SBPが常に権威的データ。NanoVDBはPhase 1-2のレンダリング用ビューに過ぎない。

移行時に変わるのはレンダリングのSDF読み取り先のみ。シェーディング側（mineral_id → LUT → アルベド等）は最初からSBPを直接読むため、移行の影響を受けない。

## 実装フェーズ

### Phase 1: NanoVDB経由レンダリング

```
- SBP → NanoVDB grid変換
- NanoVDB HDDAでsphere tracing
- 基本的なPTパイプライン確立
- 衝突判定はNanoVDB SDF queryで暫定実装
```

### Phase 2: SBPトラバーサル基盤

```
- ブリックヘッダーにsdf_min/max追加
- ブリックレベルDDA実装
- ブリック内sphere tracing実装
- NanoVDB変換を段階的に廃止
```

### Phase 3: SDF物理エンジン

```
- SDF-SDF衝突判定
- SDF直接の剛体物理（mass/inertia_tensorはper-Entityキャッシュ、決定2）
- 破壊ソルバー（決定2の拡張層）との統合
```

### 将来: 追加最適化（プロファイリング後）

```
- 階層的距離場（LODブリックとの重複を確認）
- ブリック内サブブロックスキップ
- DCエクスポートユーティリティ（需要発生時）
```
