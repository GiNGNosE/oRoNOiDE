# Stone Rendering Method — Opus Review

本ドキュメントは `stone_rendering_method.md` に対する技術レビューである。各セクションの提案手法を、独創性・リアルタイム実現性・現エンジンアーキテクチャとの整合性・実装リスクの観点から評価する。既存の実装がない手法であっても、論理的に成立するものはその旨を明示する。一方で、投資対効果（ROI）が低い方向には率直に指摘を入れる。

現エンジンの状態: SDF/voxel フィールド + Hermite サンプルによる dual contouring メッシャー、NoisyStone 編集形状、基本 PBR マテリアル（albedo, roughness, metallic, emissive）、構造不変条件バリデーション済み。

---

## 1. 石の形を大雑把に決めて、そこから細かくしていく手法

> 石の contour 太字にした時にエッジがなくなるみたいな感じで

### Review

**評価: B+ — 方向性は正しいが、記述が抽象的すぎて技術的に評価しづらい**

coarse-to-fine（粗から細へ）のアプローチ自体は、リアルタイムレンダリングにおいて最も実績のある方針の一つ。LOD 制御との親和性が高く、現エンジンの voxel 階層構造ともよく噛み合う。ここまでは良い。

問題は「contour を太字にしたらエッジがなくなる」という比喩が、技術的にどの操作を指しているのか不明瞭な点。これが SDF の平滑化（Laplacian smoothing of distance field）を意味しているなら、それは既知の手法であり、逆操作（高周波ディテールの追加）もノイズ関数やフラクタルディスプレイスメントで実現可能。ただし「関数で表現できたら当たり判定が簡単」という主張は、star-convex（重心から全表面が可視）な形状にしか成立しない。自然の石は凹部やオーバーハングを持つことが多く、単一の距離関数では表現しきれない。

**既存実装との接続:** 現在の `EditShape::NoisyStone` はまさにこの coarse-to-fine の一形態（球 + ノイズ変位）であり、方向性の妥当性はすでに部分的に実証されている。次のステップとして有効なのは、ノイズの重ね合わせを multi-octave にして地質的に妥当な表面テクスチャを生成すること。

**率直な指摘:** このセクションは着想メモとしては悪くないが、レビュー可能な「手法」にはまだなっていない。具体的なアルゴリズム（e.g., multi-resolution SDF with octave noise displacement）に落とし込まないと、実装判断の根拠にならない。

## 2. 重心から外向きに拡散モデルで点を打つ方法

> ガウスだとエッジが生まれる確率が低そう。だから、学習データが要りそう

### Review

**評価: B- — 直感は鋭いが、拡散モデルの適用範囲を過大評価している可能性がある**

ガウス過程や通常の diffusion model が滑らかなサンプルを生成しやすく、石の鋭いエッジ（劈開面、破断面）を再現しにくいという指摘は的確。これは diffusion model の帰納バイアスに由来する本質的な問題で、正しく識別できている。

ただし、ここで「だから学習データが要る」という結論に飛ぶのは少し短絡的。学習データがあればエッジが生成できるようになるかというと、それは学習アーキテクチャとロス関数の設計次第であり、データだけでは解決しない。例えば Point-E や Shap-E のような 3D 拡散モデルは大量のデータで学習されているが、鋭い幾何学的エッジの再現は依然として弱い。

**論理的に可能だが現時点で未実証な部分:** 重心から放射状にサンプル点を生成する拡散プロセスで、劈開面のような不連続な法線変化を生成させること自体は、条件付き生成（鉱物タイプ等を条件に入れる）と適切な表現空間（点群よりも SDF/implicit の方が有利）を組み合わせれば、論理的には可能。ただし、これを実装して石のリアルタイム生成に使った前例は知られていない。研究投資としてはリスクが高い。

**現エンジンとの整合性:** 拡散モデルの出力をそのまま権威的状態として使うのは、現エンジンの構造不変条件（parent/child linkage, seam coherence, version parity）と衝突する。使うなら「オフラインで生成 → SDF/voxel に変換 → エンジンに取り込む」というパイプラインに限定すべき。リアルタイムに拡散モデルを走らせてゲーム内で石を生成する構想は、現世代の GPU では推論コストの面で非現実的。

## 3. 学習データの設計

> cros pol で表面分布をとる、鉱物の種類、産地、硬さ等

### Review

**評価: A- — メタデータ設計の思考は非常に優秀。ただしスケーラビリティの壁を甘く見ている**

学習データに含めるべき属性の選定は的確。特に「産地による風化・摩耗パターンの違い」に着目している点は、地質学的に正しい洞察。川辺の石は水流による研磨で丸みを帯び、山岳の石は劈開や凍結破砕で角張る。この環境コンテキストを学習条件に含める発想は、単なるテクスチャ生成を超えた構造的リアリズムにつながりうる。

**冷静な評価:**

- Cross-polarization（cros pol）による鉱物マッピングは、薄片作成が必要な場合がほとんどで、1 サンプルあたり 30 分〜数時間のコスト。拡散モデルを訓練するには最低でも数千〜数万サンプルが必要。このスケールでの cross-pol データ収集は、研究機関レベルのリソースが必要になる。
- 「硬さを定量的に測りたい」は正しい方向だが、モース硬度は序数尺度（比率尺度ではない）であり、ビッカース硬さ試験は破壊的。非破壊で定量化するなら反発硬さ（シュミットハンマー等）が現実的だが、精度は限定的。
- 石の種類の多様性については、地球上の岩石は大きく分けて火成岩・堆積岩・変成岩の 3 類、細分すると数百種だが、ゲームで視覚的に区別可能なのはせいぜい数十カテゴリ。ここを割り切れればデータ収集の負荷は大幅に下がる。

**実用的な提案:** 学習データをゼロから収集するのではなく、既存の地質データベース（USGS rock library、British Geological Survey 等）の 3D スキャンや写真測量データをベースにし、cross-pol は検証用のゴールドスタンダードとして少数使う方が ROI は高い。

## 4. 破壊シミュレーションの学習

> 石を割るのはいいけど、元に戻らないから、石一個につき一回

### Review

**評価: B+ — 問題の本質を正確に捉えている。解決策の方向性にもう一歩踏み込みが欲しい**

「破壊は不可逆だからデータ収集が困難」という指摘は、物理ベース破壊シミュレーションの学習における根本的なボトルネックを正しく識別している。これは学術研究でも解決されていない課題。

「何でどこをどの程度の力で殴ったら、どの角度に割れたか」を全て記録するという発想は、理想的だが組み合わせ爆発に直面する。ただし「殴る物質と力をコントロールして数種類に限定する」というアイデアは、実験計画法的に正しいアプローチ。

**見落とされている選択肢:**

1. **FEM/XFEM シミュレーションによる合成データ生成:** 実際に石を割る代わりに、弾性定数と破壊靭性から有限要素法で破壊パターンを生成する方法。物理的に妥当な破壊データを大量に生成でき、パラメトリックに条件を変えられる。計算コストは高いがデータ収集コストとのトレードオフ。
2. **Voronoi 破壊 + 物性制約:** ゲーム業界で広く使われる Voronoi 分割ベースの破壊を、鉱物の劈開面方向で制約する方法。物理的厳密性は低いが、視覚的には十分説得力があり、学習データは不要。
3. **ハイブリッド:** Voronoi 破壊をベースにして、鉱物タイプごとの破壊パターンの「味付け」だけを少量データから学習する。これが最も ROI が高い。

**現エンジンとの接続:** 既存の `EditMode::Carve` と `EditCriticality::Hard` の組み合わせが破壊操作の骨格になりうる。ここに破壊パターン生成ロジックを接続するのが自然な拡張パス。

## 5. ホログラム的に石を描く

> 石の外側の決まった点から、石を描こうとする。一番難しそう

### Review

**評価: C+ — 発想は面白いが、現時点で追求する理由が薄い**

「外側の点から石を再構成する」というアプローチは、NeRF (Neural Radiance Fields) や Gaussian Splatting の文脈で近年急速に発展しているが、いずれも以下の問題を持つ:

- トポロジーの明示的な管理が困難（ボリュームが「中身」を知らない）
- 編集・破壊操作との統合が極めて難しい
- リアルタイム推論のコストが高い（特に NeRF）

著者自身が「一番難しそう」と評価しているのは正確。そして難しいだけでなく、**現エンジンの方針（構造的データが権威、メッシュはプロキシ）と根本的に相性が悪い。** ホログラム的再構成は「見た目」を再現するが「構造」を持たないため、破壊後の断面表現やマテリアル場との整合が取れない。

**唯一の活路:** オーサリングツール（石の外観リファレンスの取得）としてのみ使い、生成結果を SDF に変換してエンジンに取り込む。ただし、それなら写真測量 (photogrammetry) の方が枯れた技術で同じことができる。

## 6. 学習データの大量生成の困難さ

> 人を雇う？自動化？cros pol の律速

### Review

**評価: B — 問題認識は正しいが、解決策の探索が浅い**

データパイプラインがボトルネックになるという認識は完全に正しい。ただし「人を雇う」「自動化する」は解決策というより問題の言い換えに近い。

**もう少し具体的に考えるべき点:**

- **何のデータが本当に必要か？** 拡散モデルを訓練するための大量データが必要という前提自体を疑うべき。前述の通り、手続き的生成（procedural generation）+ 物理ベースパラメトリゼーションで、学習データなしに多様な石を生成するパスがある。学習データが必須なのは「見た目のリアリズム」を写真品質まで引き上げたい場合のみ。
- **段階的に品質を上げる:** 初期は procedural + 数十個の実データでキャリブレーション → ゲームプレイ検証 → 必要に応じてデータ拡充、が最も無駄の少ないパス。
- **既存データセットの活用:** 地質学分野には公開されている 3D 岩石スキャンデータが存在する。ゼロから収集するよりこれらを利用する方が桁違いに効率的。

## 7. 表面を関数で表現する方法

> 重心からの距離の関数で表面の contour を記述

### Review

**評価: B+ — 数学的に正しい問いを立てている**

これは球面調和関数 (Spherical Harmonics) やラジアル基底関数 (RBF) による形状表現に直結する問い。学術的には十分に研究されており、以下の手法が存在する:

- **Spherical Harmonics:** 重心からの距離を球面上の関数として展開。低次で大まかな形状、高次で細部を表現。ただし star-convex 形状にしか適用できない（凹部が表現不可）。
- **RBF interpolation:** 任意の点集合に対してスムーズな距離場を構築。凹形状も表現可能だが、制御点数に対して O(n^2) のコスト。
- **SDF (Signed Distance Field):** 現エンジンが既に採用している手法。グリッドベースの暗黙関数表現であり、任意のトポロジーを扱える。

「seam の接続が難しい」という懸念は正確。特に球面調和関数の場合、複数の石を接合する際に関数の不連続が生じる。SDF の場合は min/max 演算や smooth union でシーム処理が可能。**現エンジンの SDF + dual contouring パイプラインは、この問題に対する実質的な回答をすでに持っている。**

## 8. 当たり判定の LOD

> 離れている時は球、近づいたら細かい contour

### Review

**評価: A — 本ドキュメント中で最も実装価値の高いアイデア**

このセクションの思考は極めて実践的かつ論理的。以下の点が特に良い:

1. **LOD 階層の発想自体:** 遠距離で球/AABB、中距離で convex hull、近距離で高解像度 SDF という段階的な精度は、物理エンジンの標準的なベストプラクティスに合致する。PhysX の broad phase → narrow phase もこの思想。これを「独自に思いついた」のであれば、直感が正しい方向を向いている証拠。
2. **メモリとのトレードオフの認識:** 「全物体で細かい座標を持つのは恐ろしい情報量」→「でも三角形メッシュも同じことをしている」という自問自答は的確。実際、SDF のスパース格納（octree + 量子化距離値）は三角形メッシュより空間効率が良いケースが多い。
3. **地面との接地面の LOD 遷移問題:** これはゲーム物理で実際に発生する問題で、Havok や Bullet でも sleep/wake の遷移時にジッターが起きる。「LOD 遷移で倒れるような石は風でも倒れる」という割り切りは、ゲームデザイン的に健全な判断。

**改善の余地:**

- 「当たり判定はそんなに頻繁に起こるものじゃない」という認識は、静的な風景の石には当てはまるが、プレイヤーが投げた石や破壊片には当てはまらない。動的オブジェクトの衝突頻度を過小評価しないよう注意。
- 接地面の LOD 遷移には、ヒステリシス（一度細かくしたら一定時間は粗くしない）を入れるのが定石。

## 9. 柔らかい物質の変形（指の潰れ等）

> 手で石を持った時に指が少し潰れるのをどう再現するか

### Review

**評価: B — 良い問題提起だが、スコープを広げすぎている**

石のレンダリング手法の検討中に、手の軟組織変形まで議論を広げるのは思考の射程の広さを示しているが、同時にフォーカスの散漫さでもある。

技術的には、指の変形は position-based dynamics (PBD) や linear blend skinning + corrective blendshape で一般的に処理される。石を持った時の指の変形は、石の形状に応じた接触点でのスキン変形として処理するのが実用的で、石側に柔軟性を持たせる必要はない。これはアニメーション/キャラクターの問題であり、石のレンダリング手法とは別のレイヤーで解決すべき。

## 10. 衝突時の物理シミュレーション

> 弾性、破壊閾値、力の伝わり方

### Review

**評価: B+ — 妥協の必要性を正しく認識している点が良い**

「跳ね返す・吸収する・破壊される」の 3 分岐でハンドリングするという提案は、実はゲーム物理の標準的なアプローチにかなり近い。これを「妥協」と呼んでいるが、むしろこれが**実用的な正解**に近い。

**補足すべき点:**

- **弾性衝突:** 反発係数 (coefficient of restitution) で処理。石同士なら 0.2〜0.5 程度。計算は軽い。
- **吸収（非弾性衝突）:** 運動エネルギーの一部を熱や音に変換。これも反発係数で暗黙的に処理される。
- **破壊:** ここだけが計算コストが高い。だが、破壊イベントは稀なので、発生時のみ重い計算を走らせる設計にすれば、リアルタイム制約と両立する。

「ギミック（事前計算された破壊パターン）を使わずに正当な方法で」という姿勢は、エンジンの哲学として理解できるが、完全にリジェクトする必要もない。**事前計算 + ランタイム摂動（pre-fractured mesh に物理ベースの選択ロジックを乗せる）** は、見た目の説得力と計算コストのバランスが良く、多くの AAA タイトルで採用されている。これを「ギミック」と切り捨てるよりは、物理的な根拠を持たせた上で活用する方が現実的。

### 推奨される優先順位

1. **当たり判定 LOD の設計・実装** — 最も投資対効果が高く、現エンジンとの親和性も最高。SDF ベースの階層的衝突判定は、dual contouring パイプラインの自然な拡張になる。
2. **Coarse-to-fine 形状生成の具体化** — 現在の `NoisyStone` を multi-octave ノイズ + 鉱物タイプ別パラメトリゼーションに拡張する方向。学習データ不要で始められる。
3. **衝突物理の 3 段階モデル** — bounce/absorb/break の判定ロジックは、既存の edit system (`EditCriticality::Hard`) と自然に接続する。
4. **学習データスキーマの設計（設計のみ）** — 将来的な ML 統合を見据えてスキーマだけ先に固める。収集は後回しで良い。
5. **拡散モデル/ホログラム的手法** — 長期的な研究テーマとしては面白いが、現時点で実装リソースを割く段階にない。

### ドキュメント全体への所感

本ドキュメントは、石のレンダリングと物理シミュレーションに関するブレインストーミングの記録として価値がある。特に、問題を正しく分解して本質的な難所を識別する能力が一貫して高い（「ガウスだとエッジが出にくい」「破壊データは不可逆」「LOD 遷移で石が倒れる」等）。

一方で、改善すべき点は**具体性の不足**。多くのセクションが「こうしたらいいんじゃないか？」で終わっており、「具体的にどのアルゴリズムを、どの表現空間で、どの計算コストで実現するか」への踏み込みが弱い。次のステップとしては、上記の優先順位 1〜3 のいずれかについて、擬似コードレベルの設計ドキュメントを書くことを推奨する。

もう一点、ML（拡散モデル等）への期待が全体的に高いが、現エンジンの強みは**手続き的かつ物理ベースの構造管理**にある。ML は「味付け」や「パラメータ提案」として補助的に使うのが最も効果的であり、権威的な世界状態を ML に委ねるのは現アーキテクチャの強みを殺すことになる。この優先順位の認識が、今後の設計判断の要になる。
