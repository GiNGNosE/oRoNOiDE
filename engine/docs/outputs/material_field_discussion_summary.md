## 1. 議論の前提・批判の出発点

### 03_material_field_unified.md への批判

- メモリはボトルネックではない（48GB VRAM、14B/voxel は問題なし）
- 実装到達可能性を優先すると、設計の取り返しのつかない失敗を招く
- 構造的・描画的リアリティ最大化が絶対目標
- ROI レポートは 00_rule_before_designing の前提に反する

### SoA の「非交渉事項」への異議

- AoS の方が PT レンダリングに有利な可能性
- 格納フィールドが増えると AoS 寄りに平衡点が移動
- プロファイリングで判断すべき（API 設計で後から変更可能）

## 2. コアフィールドの現時点での落とし所

| フィールド | 判断 | 備考 |
|-----------|------|------|
| SDF | コア必須 | 形状の権威的定義 |
| 法線 | コア必須 | SDF int8 量子化のため格納 |
| mineral_id | コア必須 | 全サブシステムの起点 |
| porosity | コア | 空間変動、LUT 不可 |
| weathering_degree | コア | 化学風化の記録（色、パティナ、粗さベース） |
| microcrack_density | コア（構造専用） | 不可逆な内部損傷、レンダリング非参照 |
| crystal_orientation | **格納** | プロシージャルより格納がリアリティの天井が高い。snorm8x2 vs snorm16x2 は後で検討 |
| volume_fraction | **検討中** | 0.25mm では単一鉱物が物理的に正しい。2 鉱物（primary + secondary + blend）を検討。PT で離散境界は自然にアンチエイリアシング |
| grain_size | LUT 導出 | mineral_id LUT + per-Entity modifier |
| surface_disruption | **新規追加** | 機械的表面破壊（化学風化と分離）。蓄積深度計 + 視覚的傷の強さ |
| scratch_template_id | **新規追加** | 16 タイプ + 長さバリエーション（4bit+4bit または 8bit LUT） |
| scratch_rotation | **新規追加** | 8bit、256 方向 |

**推定コア層: 12-17 bytes/voxel**（volume_fraction の要否で変動）

## 3. サブボクセルスクラッチ（SVS）設計

- **問題**: 0.25mm 以下の傷は SDF で表現不可。プレイヤーが意図的につける傷は現実に存在する
- **解決**: テンプレート + 方向 + 蓄積 → SDF 昇格
  - scratch_template_id + scratch_rotation: 傷のパターンと方向
  - surface_disruption: 蓄積深度。閾値超過で SDF に溝を刻む（昇格）
  - 複数回同一方向に傷 → 蓄積 → 文字彫刻などに SDF 昇格で対応
- **複数方向の傷**: クロスハッチテンプレートへの遷移、多数回で等方的高粗さへ

---

## 4. ユーザー知覚とメモリ解放

- **ユーザーが記憶しているかが重要**
- **観察モード**: SVS の傷をつけられるのはこのモードのみ
- **観察モード内の傷**: 長期的に残す（高保持優先）
- **ユーザー操作 vs システム操作**: メモリ解放・LOD で差をつける
  - ユーザー操作の痕跡: 高保持、永続化
  - システム由来（風化等）: 粗い LOD で集約・解放可能
- **brick_header.user_modified**: ユーザー変更ブリックの保持優先

---

## 5. ゲーム操作と品質要件

- **Qo（観察）= max**: 最初のマイルストーンは PT による完璧な石の見た目
- **打撃・配置・破壊**: 優先度 2。チッピング（完全に割れない損傷）必須
- **打撃時の震え**: 剛体レベルの減衰振動 + モーダル解析。per-voxel シミュレーション不要
- **彫刻・研磨**: 白い傷の物理的再現。weathering_degree と surface_disruption の分離で対応

---

## 6. ブリック・メモリレイアウト

- **ブリックサイズ 8^3**: スイートスポット。L1 キャッシュに収まり、スパース効率も良好
- **SoA vs AoS**: プロファイリングで判断。API 設計で後から変更可能
- **レイアウト変更**: ブリック内部実装の変更のみで可能。取り返しのつかない決定ではない

### プロファイリング方針

プロファイリングのために別途何かを作る必要はない。最小限の PT（単純な SDF 球体 + 基本的なレイマーチング + 最小限のシェーディング：mineral_id → LUT → 色を返すだけ）は Phase 1 の最初のマイルストーンそのものであり、本番実装の最初のステップがそのままプロファイリングの基盤になる。Phase 1 では NanoVDB 経由のためブリック内レイアウトは直接影響しないが、Phase 3 での GPU 直接アクセス移行時に同じシーンで AoS/SoA を切り替えて比較できる。
