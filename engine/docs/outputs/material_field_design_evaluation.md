# マテリアルフィールド設計: 現実的評価レポート

**参照元:**
- `engine/docs/discussion/02_material_field_discussion.md` — 候補 A/B/C、MECE/SRP 分析、論点一覧
- `engine/docs/outputs/material_field_roi_recommendation.md` — ROI 駆動の 7B 推薦案
- 外部レビュー（PDF）— 候補 C ベースの保守的設計案（~14B）

**目的:** 三つの競合する設計ポジションを、プレイヤー知覚・実装リスク・物理的正当性の三軸で率直に評価し、ハイパーリアルな石のリアルタイムレンダリングを**現実的に達成する**ための設計判断を下す。

---

## 1. 問題の本質: 何が本当の制約か

### 1.1 メモリは制約ではない

議論の多くがバイト数の削減に注力しているが、率直に言えば**メモリは本プロジェクトのボトルネックではない**。

| 構成 | 28M voxels | 48GB VRAM に対する使用率 |
|------|:----------:|:----------------------:|
| 7B (ROI 推薦) | 196 MB | 0.4% |
| 10B (候補 A) | 280 MB | 0.6% |
| 13B (候補 C) | 364 MB | 0.8% |
| 14B (PDF レビュー) | 392 MB | 0.8% |

いずれの構成でも VRAM 使用率は 1% 未満。200M ボクセルの大規模シーンでも最大 2.8GB（13B 構成）であり、48GB VRAM の 6% に過ぎない。

**7B と 13B の差は 28M ボクセルで 168MB** — これはテクスチャ 1 枚分程度の差であり、設計判断を支配すべき数字ではない。

### 1.2 本当の制約

設計判断を支配すべき制約は以下の三つである。

| 制約 | 説明 | 影響 |
|------|------|------|
| **実装到達可能性** | Phase 1-2 で動作するシステムに到達できるか | 過度に洗練された設計はいつまでも動かない |
| **プロシージャルシステムの信頼性** | 格納を省略した代わりに導入するプロシージャル関数が、全ケースで破綻しないか | 破綻すると代替手段なしで品質が崩壊 |
| **アーキテクチャの回復可能性** | 初期の設計判断が間違っていた場合、フィールドの追加/削除でリカバリできるか | SoA レイアウトなら追加は容易。削除は困難 |

### 1.3 評価の原則

本レポートでは以下の原則に基づいて各フィールドを評価する。

1. **プレイヤーが知覚できない差は存在しない。** 理論的に正確でもプレイヤーが見分けられないなら、それは実装コストに見合わない。
2. **動くシステムは完璧な設計に勝る。** Phase 1 で石がレンダリングされることの方が、13B のフルスペックレイアウトの存在より価値がある。
3. **間違った賭けから回復できる設計が正しい設計。** フィールドを後から追加できることは、最初から全てを入れることより重要。
4. **プロシージャル導出は「無料」ではない。** メモリコストがゼロでも、実装コスト・デバッグコスト・エッジケースリスクがある。

## 2. 合意事項: 争いのない 6 フィールド

以下のフィールドは三つのポジション全てで合意されており、議論の余地がない。

| フィールド | 型 | サイズ | 合意根拠 |
|-----------|-----|:-----:|---------|
| **SDF** | int8 + header | 1B | 形状の権威的定義。代替不可能 |
| **法線** | snorm8×2 | 2B | SDF int8 量子化により勾配精度が不足。決定 1c の依存事項として確定済み |
| **mineral_id** | uint8 | 1B | 鉱物同一性。全サブシステムの出発点。代替不可能 |
| **porosity** | uint8 | 1B | 空間変動する空隙率。LUT では表現不能な勾配（表面 vs 内部）が存在 |
| **weathering_degree** | uint8 | 1B | 不可逆な表面変化の唯一の記録。色変化、粗さ、薄膜干渉の全てを駆動 |
| **microcrack_density** | uint8 | 1B | 不可逆な内部構造損傷の記録。破壊力学の閾値低下に不可欠 |

**合意コア: 7 bytes/voxel**

この 7B は全てのポジションの共通部分集合であり、ここに疑問の余地はない。

## 3. 争点分析: フィールドごとの評価

### 3.1 crystal_orientation — プロシージャル vs 格納

**争点の核心:** 結晶軸方向を per-voxel で 2B 格納するか、Voronoi ベースのプロシージャル関数で導出するか。

#### 各ポジション

| ポジション | 判断 | コスト | 根拠 |
|-----------|------|:-----:|------|
| ROI 推薦 | プロシージャル | 0B | 連続精度が量子化より高品質。Voronoi でグレインパターンが自動出現 |
| PDF レビュー | per-voxel 格納 | 2B | ボクセルサイズ ≈ 結晶粒サイズのため per-voxel 分解能が必要 |
| 議論文書 | 未決定（案 1/3 を提示） | 2B | レンダリング（異方性反射）と物理（劈開面）の両方で使用 |

#### 率直な分析

**ROI 推薦の主張は論理的に正しい。** Voronoi 関数は連続値を返し、65,536 方向（snorm8×2）の量子化ノイズがない。スペキュラハイライトのバンディングが消える点で、格納値より高品質であることは事実。

**しかし、ROI 推薦が過小評価しているリスクがある。**

1. **generation_transform の継承**: 破壊時に親から子へ座標変換を正しく継承しなければ、破断面の両側で結晶方位が不連続になる。これは単純なバグではなく、破壊パイプライン全体のアーキテクチャに関わる制約。Phase 3（破壊実装）まで検証できない設計依存を Phase 1-2 で確定させることになる。

2. **Voronoi の限界**: Voronoi ベースのグレインモデルは等軸晶（花崗岩等）に適するが、片状晶（雲母の葉状組織）や繊維状晶（角閃石）では自然なパターンを生成しない。全ての鉱物種に単一の Voronoi 関数で対応できるかは自明ではない。

3. **デバッグの困難さ**: 格納値は読めば即座に確認できる。プロシージャル値は「正しいはずの関数」を信じるしかない。グリントの方向がおかしい場合、原因がシェーダにあるのか Voronoi 関数にあるのか generation_transform にあるのかの切り分けが困難。

**一方で、PDF レビューの「ボクセルサイズ ≈ 結晶粒サイズ」という主張にも弱点がある。**

花崗岩の結晶粒は 1-10mm、ボクセルは 0.25mm。つまり 1 つの結晶粒は 4-40 ボクセルにまたがる。この場合、同一結晶粒内の全ボクセルが同じ crystal_orientation を持つ必要があるが、生成時にどうやってそれを保証するか。プロシージャル関数なら Voronoi セル内の全点が自動的に同じ方位を返す。格納値の場合、生成パイプラインが Voronoi 計算を行い結果を書き込む必要がある — つまり、**格納する場合でもプロシージャル生成は必要であり、その生成結果をキャッシュするかどうかの問題に帰着する**。

#### 判定

**Phase 1-2: プロシージャル導出を採用する。ただし、per-voxel 格納へのフォールバック経路を設計段階で確保する。**

根拠:
- 格納する場合でも生成時に Voronoi 計算が必要であり、プロシージャル関数は**いずれにせよ実装が必要**
- プロシージャルで始めてうまくいかなければ、SoA レイアウトなら 2B のチャネルを後から追加するだけで格納に切り替えられる（回復可能）
- 逆に、最初から格納して後からプロシージャルに切り替えることは難しい（削除は追加より困難）
- 56MB（28M × 2B）の節約は副次的。主たる理由は連続精度の品質と、グレインパターン自動生成の利便性

**Phase 3（破壊実装時）に generation_transform の継承テストを必須マイルストーンとする。** このテストに失敗した場合、per-voxel 格納に切り替える。

**リスク評価: 中。** Voronoi が不十分な鉱物種が見つかる確率は低くない。ただし回復コストも低い。

### 3.2 volume_fraction — 格納 vs 排除

**争点の核心:** 複数鉱物の混合比を per-voxel で格納するか、適応的細分化に依存して排除するか。

#### 各ポジション

| ポジション | 判断 | コスト | 根拠 |
|-----------|------|:-----:|------|
| ROI 推薦 | 排除 | 0B | 0.25mm では各ボクセルは物理的に単一鉱物。粒界は離散的 |
| PDF レビュー | 2 鉱物 + blend | 3B | 粒界の色帯・模様に混合比が必要 |
| 議論文書 | 未決定（3 方式提示） | 2-4B | GPU 互換性と視覚品質のバランス |

#### 率直な分析

**ROI 推薦の物理的議論は正しい。** 0.25mm ボクセルで花崗岩の粒界を見た場合、各ボクセルは確かに石英 OR 長石 OR 雲母のいずれかである。鉱物の混合比は物理的には存在しない — 粒界は原子スケールで急峻な遷移面であり、ボクセル解像度をはるかに下回る。

**しかし、ROI 推薦が見落としている実用上の問題がある。**

1. **LOD 遷移での品質崩壊**: 粗い LOD（5mm/voxel）では、1 ボクセルが 20×20×20 = 8,000 の細かいボクセルを代表する。mineral_id の多数決では少数派鉱物が消滅する。ROI 推薦はこの問題をブリックヘッダーの集約メタデータで解決するとしているが、これは**粒界近傍の全てのボクセル**に影響する問題であり、ヘッダー 2B で「ブリック全体」の混合比を記述するだけでは粒界が消えて均一な色面になる。

2. **プレイヤーが最も長時間見る距離は中距離**: 最細解像度（0.25mm）はズームアップ時のみ、粗い解像度（5mm+）は遠景。プレイヤーが石を「普通に」見る距離は 1-2mm/voxel 程度。この中間解像度で mineral_id が離散的なままだと、鉱物境界がジャギーに見える可能性がある。

3. **一方で、これは PT が緩和する問題でもある**: パストレーシングはピクセル内のサブピクセルサンプリングにより、境界を自然にアンチエイリアシングする。離散的な mineral_id 境界が PT のサンプリングで自然にブレンドされるため、ラスタライゼーションほど深刻なジャギーは発生しない。

**最も重要な観察**: volume_fraction の有無でプレイヤーが知覚する差は、**PT を前提とするなら小さい**。PT のストキャスティックなサンプリングは、粒界の離散的遷移を自然に平滑化する。volume_fraction が本当に効くのは、PT なしのラスタライゼーションのみでレンダリングする場合。

#### 判定

**Phase 1-2: 排除する。LOD 集約時にブリックヘッダーへ混合メタデータを格納する。**

根拠:
- PT パイプラインを前提とする本プロジェクトでは、離散的 mineral_id 境界のジャギーは PT のサブピクセルサンプリングで十分に緩和される
- 3B × 28M = 84MB の節約は副次的。主たる理由は**実装の単純化**。per-voxel に混合比を持つと、シェーダの全てのマテリアルルックアップが「2 鉱物の加重平均」になり、コードパスが複雑化する
- LOD 集約では、ブリックレベルの混合メタデータ（dominant + secondary + ratio）で十分

**ただし、Phase 2 のレンダリング品質検証で、中距離の粒界品質が不十分と判断された場合、secondary_mineral_id + blend_factor（2B）を追加する経路を残す。** SoA レイアウトならチャネル追加のコストは低い。

**リスク評価: 低〜中。** PT のアンチエイリアシング効果が粒界のジャギーをどこまで緩和するかは、実際にレンダリングしてみないと確定しない。しかし回復コストは低い（2B チャネル追加）。

### 3.3 grain_size — LUT 導出 vs 格納

**争点の核心:** 結晶粒径を per-voxel で 1B 格納するか、mineral_id LUT + per-Entity modifier で導出するか。

#### 各ポジション

| ポジション | 判断 | コスト | 根拠 |
|-----------|------|:-----:|------|
| ROI 推薦 | LUT 導出 | 0B | 鉱物種ごとの粒径は概ね一定。岩石間差は per-Entity modifier で表現 |
| PDF レビュー | 結論で明示的に触れず | — | 候補 C のフィールド一覧を踏襲（暗黙的に格納） |
| 議論文書 | コア層に 1B | 1B | 内在的性質、テクスチャスケール制御に使用 |

#### 率直な分析

**ROI 推薦の主張は強い。** 同一岩石内で、石英の粒径は石英なりの値、雲母は雲母なりの値を持つ。この値は mineral_id LUT に格納すれば per-voxel 格納は不要。岩石ごとの「粗粒/細粒」の違いは per-Entity の grain_size_modifier で表現可能。

**per-voxel grain_size が必要になる唯一のケースは、同一鉱物種が空間的に大きく異なる粒径を持つ場合。** 接触変成帯の再結晶、級化層理の粒径勾配などだが、これらは極めて稀。

**プレイヤーの知覚**: grain_size はテクスチャスケール（プロシージャルノイズの周波数）とグリント粒度に影響する。しかし、mineral_id LUT + per-Entity modifier で生成した粒径が「物理的に 100% 正確」でなくても、プレイヤーが「この石の粒が不自然に大きい/小さい」と感じる確率は極めて低い。

#### 判定

**LUT 導出を採用する。**

根拠:
- per-voxel grain_size の視覚的インパクトは、mineral_id + per-Entity modifier で十分に近似可能
- 1B × 28M = 28MB の節約は副次的。主たる理由は**フィールド数の削減によるシェーダの単純化**
- 稀なケース（接触変成帯）は per-Entity の低解像度 3D テクスチャ（~4KB）で補完可能

**リスク評価: 極低。** 粒径の空間変動がプレイヤーの知覚を支配するシナリオは考えにくい。

### 3.4 microcrack_density の視覚的役割

**争点の核心:** microcrack_density はレンダリングに影響するか（表面粗さ、可視クラック）、それとも構造シミュレーション専用か。

#### 各ポジション

| ポジション | 判断 | 根拠 |
|-----------|------|------|
| ROI 推薦 | 構造専用、レンダリング非参照 | マイクロクラックはμmスケール、ボクセル解像度の 1/10 以下。表面の見た目は weathering_degree が担う |
| PDF レビュー | 明示的に触れず（議論文書の定義を踏襲） | — |
| 議論文書 | V9（クラック描画）と W3（損傷蓄積）の両方をカバー | MECE 分析で両方にマッピング |

#### 率直な分析

**ROI 推薦の物理的議論は正しい。** マイクロクラックの個々のサイズは数μm〜数十μm。0.25mm（250μm）ボクセルの 1/10 以下であり、個々のクラックが表面の見た目を直接変えることはない。

**ただし、議論文書の MECE 分析も間違ってはいない。** microcrack_density が「高い」領域は、集合的に表面粗さを増加させる。これは物理的に正しい — 高密度のマイクロクラックを含む岩石表面は、触ると粗く、光学的にもマットに見える。

**問題は責務の分離。** weathering_degree は「化学的風化がどれだけ進んだか」を記述する。microcrack_density は「機械的損傷がどれだけ蓄積したか」を記述する。両者は独立な現象だが、表面粗さという同一の視覚的結果に影響する。

- 化学風化（weathering_degree↑）→ 粗さ↑（差分侵食による）
- 機械的損傷（microcrack_density↑）→ 粗さ↑（マイクロクラックによる散乱）

これら二つの経路を weathering_degree だけで代表させると、「新鮮だが内部的にボロボロの石」と「風化が進んだが機械的に健全な石」が同じ粗さに見えてしまう。

**しかし、プレイヤーがこの区別を知覚するか?** 現実的には、**知覚しない。** 粗さの微妙な差は、通常のゲームプレイ距離では判別不能。microcrack_density が高い石は「割れやすい」ことが重要であり、それは構造サブシステムが担う。

#### 判定

**microcrack_density は構造シミュレーション専用フィールドとする。レンダリングシェーダは参照しない。**

根拠:
- 表面粗さは weathering_degree + mineral_id LUT で十分にリアルに導出可能
- microcrack_density のレンダリング参照を排除することで、レンダリングシェーダが読むフィールドが 5 チャネル（6B）に減少し、メモリ帯域幅が削減される
- 可視クラック（目に見える大きなヒビ）は SDF ジオメトリ変更として表現されるべき。microcrack_density は不可視の微小損傷であり、可視クラックとは物理的に別の現象

**リスク評価: 極低。** microcrack_density が粗さに影響しないことによる視覚的品質の低下は、プレイヤーの知覚閾値以下。

## 4. 総合判定: 推奨コアレイアウト

### 4.1 推奨: 7 bytes/voxel

```
[コア層 — 全アクティブボクセル]
SDF:                 int8 + header     (1B)
法線:                snorm8×2          (2B)
mineral_id:          uint8             (1B)
porosity:            uint8             (1B)
weathering_degree:   uint8             (1B)
microcrack_density:  uint8             (1B)
──────────────────────────────────────────
合計: 7 bytes/voxel
```

### 4.2 この判断の要約

| フィールド | 判断 | 理由の一文要約 |
|-----------|------|-------------|
| SDF | コア必須 | 形状の権威的真実 |
| 法線 | コア必須 | SDF int8 量子化で勾配精度不足 |
| mineral_id | コア必須 | 全サブシステムの起点 |
| porosity | コア | 空間変動する物理量。LUT 不可 |
| weathering_degree | コア | 不可逆な表面変化の唯一の記録 |
| microcrack_density | コア（構造専用） | 不可逆な内部損傷。レンダリング非参照 |
| crystal_orientation | **プロシージャル** | 連続精度 + グレインパターン自動生成。per-voxel 格納はフォールバック |
| volume_fraction | **排除** | PT のサブピクセルサンプリングで粒界を自然に平滑化。LOD はヘッダーメタデータ |
| grain_size | **LUT 導出** | mineral_id LUT + per-Entity modifier で十分 |

### 4.3 レンダリングシェーダが読むフィールド

```
[レンダリングパス: 5 フィールド / 6B]
  SDF (1B)            → レイマーチング / メッシュ抽出
  法線 (2B)           → シェーディング方向
  mineral_id (1B)     → LUT → アルベド, F0, SSS, 粗さベース値
  porosity (1B)       → SSS 深度, 透過率
  weathering_degree (1B) → 色変化, 薄膜干渉, 粗さ修正

  + crystal_orientation (プロシージャル) → 異方性グリント, 劈開面反射

[レンダリングが読まないフィールド]
  microcrack_density → 破壊シミュレーション専用
```

### 4.4 メモリ見積もり

| シナリオ | ボクセル数 | 7B コア | 参考: 13B (候補 C) |
|---------|:--------:|:------:|:----------------:|
| 平均的な風景（石 ~300個 + 地形） | 28M | 196 MB | 364 MB |
| 大規模シーン（石 1000個 + 広域地形） | 200M | 1.4 GB | 2.6 GB |
| 操作圏 3 石のシミュレーション時 | +13M (拡張) | +260 MB | +260 MB |

## 5. 拡張層・per-Entity 設計

拡張層の設計は三つのポジション全てで実質的に一致している。本推薦でもこれを踏襲する。

### 5.1 拡張層（シミュレーション対象領域のみ、解放可能）

```
[拡張層 — 破壊/風化シミュレーション時のみ確保]
phase_field_d:       float16           (2B)
E:                   float16           (2B)
nu:                  float16           (2B)
Gc:                  float16           (2B)
sigma_t:             float16           (2B)
anisotropy_tensor:   float16×3         (6B)
moisture:            float16           (2B)
──────────────────────────────────────────
拡張: 20 bytes/voxel（対象領域のみ）
```

ライフサイクル:
1. **展開**: コアフィールド + プロシージャル crystal_orientation から力学パラメータを LUT 導出して初期化
2. **実行**: ソルバーが phase_field_d を伝播、力学パラメータを更新
3. **コミット**: 結果を microcrack_density（コア）と SDF（コア）に反映
4. **解放**: VRAM を回復。コア層だけで完璧なレンダリングが継続

### 5.2 per-Entity メタデータ

```
[per-Entity]
topology_version:      uint32           (4B)
temperature:           float16          (2B)
layering_orientation:  float16×3        (6B)
grain_size_modifier:   float16          (2B)
generation_transform:  float32×3×4      (48B)    ← プロシージャル座標安定性
──────────────────────────────────────────
per-Entity: 62 bytes/Entity
```

### 5.3 温度モデル

議論文書 Section 3.7 の 3 層温度モデルをそのまま踏襲:
- Layer 1: per-Entity temperature（常時、2B/Entity）
- Layer 2: 導出熱物性（格納なし、LUT + コアフィールドから計算）
- Layer 3: per-voxel temperature（拡張層、熱衝撃/局所加熱時のみ一時確保）

## 6. アーキテクチャ上の非交渉事項

### 6.1 SoA レイアウト（必須）

```
[SoA: Structure of Arrays]
SDF[0], SDF[1], SDF[2], ...
normal[0], normal[1], normal[2], ...
mineral_id[0], mineral_id[1], ...
porosity[0], porosity[1], ...
weathering_degree[0], weathering_degree[1], ...
microcrack_density[0], microcrack_density[1], ...
```

SoA は本設計の**回復可能性を保証する基盤**。

| 操作 | SoA | AoS |
|------|-----|-----|
| フィールド追加 | 新しい配列を追加するだけ。既存データに影響なし | 全ボクセルのストライドが変わり、全データの再配置が必要 |
| フィールド削除 | 配列を解放するだけ | 同上 |
| 部分的フィールドアクセス | 必要なチャネルのみ読み込み。帯域効率が高い | 不要なフィールドもキャッシュラインに乗る |

Phase 3-4 でプロファイリングにより AoS が有利と判明した場合の移行は検討するが、初期は SoA で設計の柔軟性を優先する。

### 6.2 LOD 集約メタデータ（ブリックヘッダー）

volume_fraction を per-voxel から排除する代わりに、粗い LOD での鉱物混合をブリックヘッダーで管理する:

```
[ブリックヘッダー — 粗い LOD のみ]
dominant_mineral_id:    uint8   (1B)
secondary_mineral_id:   uint8   (1B)
blend_ratio:            uint8   (1B)
──────────────────────
+3B per brick header（粗い LOD ブリックのみ）
```

これにより、粗い LOD でも 2 鉱物の混合レンダリングが可能。per-voxel コストはゼロ。ブリック（8^3 = 512 voxels）あたり 3B のオーバーヘッドは無視可能（0.006B/voxel 相当）。

### 6.3 プロシージャル crystal_orientation の設計要件

```
voronoi_crystal_orientation(seed_pos, mineral_id, entity_params):
  cell_size = LUT[mineral_id].grain_size × entity_params.grain_size_modifier
  cell = quantized_cell(seed_pos / cell_size)
  nearest = find_nearest_voronoi_center(seed_pos, cell)
  return hash_to_orientation(nearest.cell_id, mineral_id)
```

**座標安定性の保証:**
```
seed_pos = entity.generation_transform × voxel_entity_local_pos
```
- 生成時のワールド変換を per-Entity に記録
- 破壊時に子エンティティは親の generation_transform を継承
- 同一ボクセルは常に同じ seed_pos → 同じ orientation を返す

## 7. リスクマトリクス

| 賭け | 失敗シナリオ | 発生確率 | 影響度 | 回復コスト | 回復方法 |
|------|-----------|:-------:|:-----:|:--------:|---------|
| プロシージャル crystal_orientation | Voronoi が特定鉱物種（雲母、角閃石）で不自然なパターンを生成 | 中 | 中 | **低** | SoA に snorm8×2 チャネル（2B）を追加し、生成時に Voronoi 結果をキャッシュ |
| プロシージャル crystal_orientation | generation_transform の継承が破壊パイプラインで破綻 | 低〜中 | 高 | **低〜中** | 同上。ただし破壊パイプラインの設計変更が必要な場合は中コスト |
| volume_fraction 排除 | 中距離 LOD で粒界のジャギーが PT でも目立つ | 低 | 中 | **低** | secondary_mineral_id + blend_factor（2B）を SoA に追加 |
| volume_fraction 排除 | LOD ブリックヘッダーの混合メタデータだけでは遠景品質が不足 | 低 | 低 | **極低** | ブリックヘッダーの混合メタデータを拡張（+数 B/brick） |
| grain_size LUT 導出 | 特殊岩石（接触変成帯）の粒径変動が知覚可能 | 極低 | 低 | **極低** | per-Entity 低解像度 3D テクスチャ（16^3, ~4KB）で補完 |
| microcrack_density 非視覚化 | 衝撃後の即座の視覚フィードバック不足がプレイヤー体験を損ねる | 中 | 低 | **なし** | パーティクルエフェクト + 音響フィードバック（マテリアルフィールドの変更不要） |

**リスクの総合評価:** 全ての賭けにおいて、失敗時の回復コストが低い。これは SoA レイアウトの採用が保証する設計上の性質であり、「間違った賭けから回復できる設計」の原則に適合している。

## 8. 実装フェーズ

### Phase 1: 最小レンダリング (4B/voxel)

```
フィールド: SDF (1B) + 法線 (2B) + mineral_id (1B) = 4B

目標:
  - SBP + NanoVDB 変換でフレームループ確立
  - mineral_id → LUT でアルベド + ラフネス取得
  - 法線でシェーディング
  - 単一鉱物の石をレンダリング

検証項目:
  - フレームレート安定性
  - SDF int8 + 法線 snorm8×2 の視覚品質
```

### Phase 2: フルコア + プロシージャル (7B/voxel)

```
追加フィールド: porosity (1B) + weathering_degree (1B) + microcrack_density (1B) = +3B

追加システム:
  - プロシージャル crystal_orientation（Voronoi 関数）
  - per-Entity generation_transform
  - per-Entity grain_size_modifier
  - LOD 集約メタデータ（ブリックヘッダー）

目標:
  - 多鉱物岩（花崗岩）のレンダリング
  - 異方性グリント
  - 風化による色変化・粗さ変化
  - SSS（porosity 依存）

検証項目 ← ここが決定的:
  ☐ プロシージャル crystal_orientation のグリント品質（花崗岩、大理石、片麻岩）
  ☐ 粒界の離散的遷移の視覚品質（PT アンチエイリアシング効果の確認）
  ☐ LOD 遷移時の鉱物混合品質
  ☐ 中距離での石の全体的なリアリティ
```

**Phase 2 完了時にフィールド追加判断を行う:**
- 粒界品質が不足 → secondary_mineral_id + blend_factor を追加（→ 9B）
- Voronoi が不十分な鉱物種 → crystal_orientation を格納に切替（→ 9B）
- 両方 → 11B（それでも候補 C の 13B より 2B 少ない）

### Phase 3: 拡張層 + 破壊 (+20B/voxel, 対象領域のみ)

```
追加: 拡張層 (20B/voxel、シミュレーション対象領域のみ)

目標:
  - フェーズフィールド破壊ソルバー
  - generation_transform の継承テスト（必須マイルストーン）
  - 力学パラメータの LUT 導出 → 拡張層展開
  - microcrack_density へのコミット
  - 拡張層の展開/解放ライフサイクル
```

### Phase 4: 風化 + 環境連成

```
追加: moisture (拡張層)、per-voxel temperature (拡張層)

目標:
  - 風化エージェント
  - 温度モデル（3 層）
  - 水分浸透シミュレーション
```

## 9. 最終比較: 四つのポジション

| 評価軸 | 重み | 候補 A (10B) | 候補 C (12-13B) | PDF レビュー (~14B) | **本推薦 (7B)** |
|--------|:----:|:----------:|:-------------:|:----------------:|:-------------:|
| レンダリング忠実度 | 最高 | 中 | 最高 | 最高 | **最高** |
| 破壊正確性 | 最高 | 中 | 最高 | 最高 | **最高** |
| メモリ効率 | 低 | 高 | 中 | 中-低 | **最高** |
| Phase 1 到達速度 | 高 | 中 | 低 | 低 | **最高** |
| 間違いからの回復可能性 | 高 | 中 | 低 | 低 | **最高** |
| 永続化 | 中 | 高 | 最高 | 最高 | **最高** |
| 拡張性 | 中 | 低 | 高 | 高 | **最高** |

### 9.1 各ポジションの率直な評価

**候補 A (10B)**: volume_fraction がなく crystal_orientation もないため、レンダリング忠実度と破壊正確性が「中」に留まる。しかし、その単純さは過小評価されるべきではない。

**候補 C (12-13B)**: 理論的には最もバランスが良い。しかし、volume_fraction の設計が未決定のまま 3-4B を確保しているため、「何を格納するか」が決まる前に「格納する」と決めてしまっている。実装フェーズでは、未決定のフィールドのために全ボクセルのストライドを確保しておくことになり、開発初期の身軽さが失われる。

**PDF レビュー (~14B)**: 最も保守的で安全。全てを格納し、プロシージャルへの依存を最小化する。リスクは低いが、「必要になるかわからないフィールドにメモリを払い続ける」設計。14B × 200M = 2.8GB は依然として余裕があるが、候補 C よりさらに重い。

**本推薦 (7B)**: 合意事項のみをコアに確定し、争点はプロシージャル/導出で開始する。間違いが判明した時点でフィールドを追加する。**最大のリスクは「全部間違っていた」場合だが、その場合でも 7B → 13B への拡張は SoA レイアウト上で段階的に可能。** 最大のメリットは Phase 1-2 が最も早く動作状態に到達すること。

### 9.2 なぜ 7B で E1/E2 が「最高」と評価できるか

候補 C と本推薦でレンダリング忠実度が同等と評価する根拠:

1. **crystal_orientation**: プロシージャル Voronoi は連続精度で、snorm8×2 の量子化精度を上回る。Voronoi で自然なグレインパターンが自動出現する。格納と同等以上。
2. **volume_fraction**: PT のサブピクセルサンプリングが離散的 mineral_id 境界を自然に平滑化する。LOD ではブリックヘッダーの混合メタデータで対応。実質的な視覚差はプレイヤーの知覚閾値以下。
3. **grain_size**: LUT + per-Entity modifier による粒径制御で、格納と知覚的に同等。
4. **microcrack_density の非視覚化**: 表面粗さは weathering_degree + mineral_id LUT で十分に表現可能。マイクロクラックの視覚的寄与はプレイヤーの知覚閾値以下。

**逆に、7B が候補 C に劣る可能性があるポイント:**
- LOD 遷移時の粒界品質（volume_fraction の不在）
- 特殊鉱物種の Voronoi パターン（crystal_orientation のプロシージャル限界）
- これらは Phase 2 の検証で判明し、フィールド追加で回復可能

## 10. 結論

三つのポジションは、それぞれ異なるリスク許容度に基づいた正当な判断を行っている。PDF レビューは安全性を、ROI 推薦は効率性を、議論文書はバランスを重視している。

本レポートの判断は以下の認識に基づく:

**ハイパーリアルな石のリアルタイムレンダリングを「現実的に達成する」ために最も重要なのは、早期に動作するシステムを構築し、実際のレンダリング結果を見ながら設計を進化させることである。** 7B で開始し、Phase 2 の視覚品質検証を経て、必要に応じてフィールドを追加する — この段階的アプローチが、13B のフルスペックを最初から目指して Phase 1 の到達が遅れるよりも、最終的により高い品質に到達する可能性が高い。

メモリ差（168MB）のために設計を複雑化する理由はない。しかし同時に、「使わないかもしれないフィールドを最初から全ボクセルに確保する」理由もない。**SoA レイアウトの採用が、この二つの主張を両立させる。** 必要になったら追加する。不要とわかったら削除する。この柔軟性こそが、本プロジェクトの不確実性に対する最も現実的な回答である。

最終推薦: **7 bytes/voxel コア + SoA レイアウト + Phase 2 検証ゲート**。
